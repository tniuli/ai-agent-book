# 第19章 Agent 商业模式

> 穷则变，变则通，通则久。——《易经·系辞下》

从技术到商业，是每一个 Agent 开发者必须跨越的鸿沟。技术再强，找不到商业模式，终究只是实验室里的玩具。本章将帮你理解 Agent 作为产品 vs Agent 作为平台的两种定位和各自的商业逻辑，掌握垂直领域 Agent 的商业机会识别方法，了解 Agent Marketplace 的三种形态（封闭市场、开放市场、协议网络），并学会从 Demo 到 PMF 的精益验证方法。

## 19.1 Agent 作为产品 vs Agent 作为平台

> ⚠️ 常见陷阱：把技术 Demo 当成产品。Agent 领域最危险的幻觉是"Demo 效应"——在精心挑选的场景下，Agent 表现惊艳；但一旦面对真实用户的多样化需求，问题层出不穷。Demo 到 PMF 之间的距离，可能是 Agent 创业中最容易被低估的鸿沟。解决方案：用 Wizard of Oz 方法零成本验证需求；在写一行代码之前，先确认用户是否愿意为这个能力付费。

### 19.1.1 两种根本不同的商业逻辑

当你决定把一个 Agent 推向市场时，首先要回答一个根本性问题：你的 Agent 是一个产品，还是一个平台？

这看似简单的问题，决定了你后续所有的技术选型、团队配置和商业策略。

**Agent 作为产品**，意味着你直接向最终用户提供一个解决问题的工具。用户打开你的应用，描述需求，Agent 完成任务，用户付费。这是最直接、最容易理解的商业模式。就像你开了一家餐厅，顾客来了就吃，吃完就付钱。

**Agent 作为平台**，意味着你不直接服务最终用户，而是提供基础设施、工具链或市场，让其他人来构建和运营 Agent。你赚的是"修路钱"和"收租钱"。就像你建了一座商场，自己不卖货，但每家店都要给你交租金。

我们来对比一下两种模式的核心差异：

| 维度 | Agent 作为产品 | Agent 作为平台 |
|------|---------------|---------------|
| 核心能力 | 解决特定问题的深度能力 | 通用基础设施和生态运营 |
| 收入模式 | 订阅费、按次付费 | 平台佣金、API 调用费、广告 |
| 用户画像 | 终端用户（个人或企业） | 开发者、企业、其他 Agent 团队 |
| 护城河 | 数据壁垒、工作流深度 | 网络效应、生态锁定 |
| 技术重心 | Agent 能力本身 | 可扩展性、开放性、稳定性 |
| 典型案例 | Cursor、Jasper、Harvey | GPT Store、Coze、Dify |

### 19.1.2 Agent 产品的商业特征

做 Agent 产品，本质上做的是 SaaS（Software as a Service）的升级版——从"软件即服务"到"智能即服务"（Intelligence as a Service）。但与传统 SaaS 相比，Agent 产品有几个显著不同的商业特征：

**第一，边际成本不为零。** 传统 SaaS 产品一旦开发完成，服务一个用户和服务一百万个用户的边际成本几乎为零。但 Agent 产品不同——每一次任务执行都意味着大模型的 API 调用，这意味着真金白银的推理成本。一个用户频繁使用，你的成本可能比他的订阅费还高。

这就要求你在定价时必须精细计算：单次任务的平均 Token 消耗是多少？不同复杂度的任务成本差异有多大？你的定价是否留出了足够的毛利空间？

**第二，效果即留存。** 传统 SaaS 的留存靠的是切换成本——用户习惯了你的界面，数据都存在你这里，换一个太麻烦。但 Agent 产品如果效果不好，用户会毫不犹豫地切换，因为 Agent 的切换成本极低——换个对话框而已。

所以 Agent 产品的核心不是锁定用户，而是让用户离不开你的效果。你的 Agent 比竞品快 20%、准确率高 10%，用户就会留下来。这要求你在垂直场景上做到极致。

**第三，信任门槛极高。** 用户把决策权交给 Agent，意味着极大的信任。尤其是财务、法律、医疗等高风险领域，一次错误可能导致严重后果。你的产品必须在"可控的自主"和"高效的自动"之间找到平衡点——既不能让用户觉得事事都要确认（那和手动操作有什么区别），又不能让用户觉得 Agent 在失控地操作。

### 19.1.3 Agent 平台的网络效应

做 Agent 平台，玩的是另一套逻辑。平台的核心壁垒是网络效应——使用的人越多，平台越有价值。

Agent 平台的网络效应有两个层次：

**供给侧网络效应：** 平台上的 Agent 越多，越能吸引更多的用户来寻找解决方案；而更多的用户又吸引更多的开发者来构建 Agent。这是一个正向飞轮。GPT Store 和 Coze 都在努力启动这个飞轮。

**数据网络效应：** 平台汇聚的交互数据越多，越能优化推荐算法、提升 Agent 的质量、发现新的用户需求。这些数据还能反馈到基座模型的训练中，形成更深层次的壁垒。

但 Agent 平台也面临独特的挑战：

**质量控制的困境。** 开放平台意味着任何人都能发布 Agent，质量参差不齐。一个糟糕的 Agent 不仅伤害它自己的用户，还会损害整个平台的声誉。App Store 用审核机制解决这个问题，但 Agent 的行为更加动态和不可预测，审核难度呈指数级增长。

**标准化的两难。** 平台需要标准化来降低开发门槛，但过度标准化又限制了 Agent 的能力上限。就像 Web 的标准化带来了繁荣，但也限制了浏览器能做的事情。

### 19.1.4 如何选择：产品还是平台？

对于大多数初创团队，我建议先做产品，再考虑平台。原因很简单：

**做产品验证速度更快。** 你可以在几周内验证一个 Agent 产品是否有真实需求，但平台需要足够的双边市场才有效，验证周期至少半年。

**做产品的技术门槛更低。** 你只需要把一个场景做深做透，而平台需要解决通用性、扩展性、安全性等一系列难题。

**做产品的资金效率更高。** 平台需要大量前期投入来建设基础设施和吸引双边市场，而产品可以用精益的方法逐步验证和迭代。

但如果你已经拥有某种平台级资源——比如大量的开发者社区、独有的数据资产、或者强大的分发渠道——那么直接做平台也是合理的选择。

一个常见的进化路径是：**先做一个出色的 Agent 产品，然后在产品的基础上逐步开放能力，吸引第三方开发者，最终演变为平台。** 这就是 Slack、Stripe 走过的路，也是很多 Agent 公司正在走的路。

## 19.2 垂直领域 Agent 的商业机会

### 19.2.1 为什么要做垂直？

通用 Agent 的战争，是大厂的游戏。OpenAI、Google、Anthropic 这些拥有顶级基座模型的公司，天然在通用 Agent 领域占据优势。对于创业公司和独立开发者来说，做通用 Agent 几乎没有胜算。

但垂直领域是完全不同的战场。在垂直领域，决胜的关键不是模型有多大，而是你对场景的理解有多深、你的数据有多独特、你的工作流有多贴合用户需求。

垂直 Agent 的商业逻辑可以用一个公式概括：

**价值 = 替代成本 × 使用频率 × 场景独特性**

- **替代成本**：Agent 替代的人工有多贵？替代律师的 Agent 比替代客服的 Agent 价值高一个数量级。
- **使用频率**：用户多久用一次？每天用的 Agent 比每月用的 Agent 有更强的付费意愿。
- **场景独特性**：这个场景的数据和知识有多稀缺？越稀缺，壁垒越高。

### 19.2.2 六大高价值垂直领域

根据上述公式，我筛选出六个最具商业价值的垂直领域，并逐一分析。

**1. 法律 Agent**

法律行业的替代成本极高——律师的小时费率通常在 500-3000 元之间。法律文档的审查、合同的起草、法规的检索，都是高度结构化且重复性的工作，非常适合 Agent 自动化。

Harvey 是法律 Agent 领域的标杆，已经获得了超过 2 亿美元的融资。它的成功证明了法律 Agent 的商业可行性。

法律 Agent 的壁垒在于：
- 法律知识的专业性和时效性（法规持续更新）
- 数据的机密性要求（客户-律师特权）
- 准确性的极端要求（一个条款的错误可能导致巨额损失）

**2. 医疗 Agent**

医疗行业的替代成本同样极高，但监管门槛也最高。医疗 Agent 的切入点不是替代医生，而是辅助医生——从病历整理、影像初筛、用药检查到患者随访。

医疗 Agent 的独特壁垒在于：
- 医疗数据的隐私保护（HIPAA 等法规）
- 临床验证的长期性
- 医疗责任的模糊地带

但正因为门槛高，一旦建立壁垒，竞争对手很难跟进。

**3. 金融 Agent**

金融行业是数据最密集、决策最频繁的行业之一。从投资研究、风险评估、合规检查到客户服务，几乎每个环节都有 Agent 的用武之地。

金融 Agent 的核心壁垒是实时数据的获取和处理能力，以及风控模型的准确性。一个优秀的金融 Agent 需要在毫秒级别做出判断，这要求极低的延迟和极高的可靠性。

**4. 教育 Agent**

教育是少数几个"高频 + 高替代成本 + 巨大市场"同时满足的领域。一个一对一的私人导师每小时收费数百元，而 Agent 可以以极低的边际成本提供接近一对一的教学体验。

教育 Agent 的关键挑战是教学效果的量化——你怎么证明你的 Agent 比传统的学习方式更有效？这需要长期的教育实验和数据积累。

Khan Academy 的 Khanmigo 是教育 Agent 的先驱之一，它展示了 Agent 在个性化教学方面的巨大潜力。

**5. 代码 Agent**

代码 Agent 是目前商业化最成功的垂直领域之一。Cursor、GitHub Copilot、Devin 等产品已经证明了开发者愿意为提效工具付费。

代码 Agent 的优势在于：
- 开发者对新技术接受度高
- 效果可量化（代码提交量、Bug 率、开发周期）
- 数据天然结构化（代码本身就是结构化数据）
- 开发者社区的自传播效应

**6. 销售 Agent**

销售是企业的生命线，也是人力成本最集中的环节之一。从线索筛选、客户触达、需求分析到方案推荐，Agent 可以在销售的每个环节提供自动化支持。

销售 Agent 的独特价值在于它可以 7×24 小时不间断工作，处理大量并发的客户对话，而且永远保持"最佳状态"——不会因为心情不好而影响客户体验。

### 19.2.3 垂直 Agent 的进入策略

选定领域后，如何切入？我总结了三种常见的进入策略：

**痛点切入法：** 找到目标用户最痛的一个具体问题，用 Agent 解决它。比如法律领域，不要一上来就做"AI 律师"，而是先做"合同审查 Agent"，解决律师花大量时间审阅合同这个具体痛点。做深一个点，再逐步扩展到相邻场景。

**数据切入法：** 如果你拥有某个领域的独特数据资产，直接从数据出发构建 Agent。比如你有一个法律案例数据库，就可以基于这个数据库构建法律检索 Agent，你的数据壁垒就是天然的护城河。

**工作流切入法：** 不做最智能的 Agent，而是做最懂工作流的 Agent。深入理解目标用户的操作流程，把 Agent 无缝嵌入现有工作流中。用户不需要改变习惯，只是某些步骤从手动变成了自动。这种方式用户接受度最高，切换成本也最高。

### 19.2.4 深度案例：一个法律合同 Agent 从 0 到 PMF 的全过程

理论讲了很多，让我们看一个具体的案例——一个虚构但基于真实行业逻辑的法律合同审查 Agent "ContractGuard"，从想法到 PMF 的完整过程。

**第 0 个月：发现痛点。** 创始人张律师在一家律所工作，每天花 3-4 小时审查 NDA（保密协议）。NDA 的审查高度重复——90% 的条款是标准模板，只有 10% 需要仔细斟酌。但律师必须逐条阅读，因为那 10% 可能藏着重大风险。张律师想：能不能让 AI 先筛一遍，我只需要关注它标记的风险条款？

**第 1 个月：最小验证。** 张律师用 GPT-4 API 写了一个简单的 NDA 审查脚本——输入合同文本，输出风险条款列表和修改建议。拿 50 份已审查过的 NDA 做回测，发现召回率只有 60%：很多风险条款没标出来，也标了不少误报。但 60% 已经足够说明方向可行——问题出在提示词和领域知识不足，不是技术路线错误。

**第 2-3 个月：领域知识注入。** 张律师和另一位合伙人整理了 200 份标注过的 NDA（每份标注了风险条款、风险等级、修改建议），用这些数据做了三件事：① 构建了 RAG 知识库，让 Agent 在审查时可以参考历史案例和风险条款库；② 重写了系统提示词，从通用的"审查合同"改为结构化的 NDA 审查清单（包含 30+ 个常见风险点和检查规则），让 LLM 按清单逐项检查而非自由发挥；③ 建立了风险等级体系（高/中/低），让输出更结构化。回测召回率（Recall，即实际风险条款中被 Agent 成功标出的比例）从 60% 提升到 82%。

**第 4 个月：种子用户。** 张律师在律师社群发布了 ContractGuard 的内测版，获得了 30 个种子用户。关键反馈：① 用户不关心 Agent 用了什么模型，只关心"它漏报了多少风险条款"；② 用户需要和现有工作流集成——他们用 Word 审合同，不想切换到新工具；③ 82% 的召回率不够——律师的容忍度是"漏报率接近零"，误报可以多一些，但漏报不可接受。

**第 5-6 个月：调整方向。** 基于反馈做了三个关键调整：① 从"审查所有条款"改为"只标记风险条款"——降低误报，让律师只看 Agent 标红的部分；② 开发 Word 插件，让 Agent 直接在 Word 中标注风险，无需切换工具；③ 引入"人工复核"环节——Agent 标记风险后，律师确认或排除，确认的风险条款进入知识库，形成数据飞轮。召回率提升到 91%，漏报率降到 5% 以下。

**第 7-8 个月：定价与付费。** 推出付费版：免费版每月审查 5 份 NDA，专业版 299 元/月不限量。30 个种子用户中，12 个转为付费用户。付费用户的平均使用频率是每周 8 份 NDA，替代了约 4 小时/周的人工审查时间。NDA 审查通常由初级律师或律师助理完成，小时费率约 300-500 元，每月节省约 4,800-8,000 元。299 元的定价有 16-27 倍的价值感知——用户觉得"赚了"。

**第 9 个月：PMF 信号。** 出现了三个 PMF 信号：① 付费用户留存率超过 85%；② 用户主动推荐同事使用，口碑获客占比超过 40%；③ 用户开始要求扩展到其他合同类型（劳动合同、租赁合同）。这标志着 ContractGuard 从"一个 NDA 审查工具"走向"一个合同审查平台"。

**复盘关键决策：**

| 决策 | 选择 | 另一个选项 | 为什么这样选 |
|------|------|-----------|-------------|
| 切入点 | NDA 审查 | AI 律师 | NDA 足够结构化、重复性高，容易做出 60% 的初始效果 |
| 产品形态 | Word 插件 | 独立 Web 应用 | 律师的工作流在 Word 中，切换成本是最大阻力 |
| 核心指标 | 漏报率 | 整体准确率 | 律师的痛点是"漏看风险"，不是"多看了几条" |
| 数据策略 | 人工复核 + 数据飞轮 | 纯自动化 | 法律场景不能容忍 5% 的漏报，人工复核是安全网也是 RAG 知识库的数据来源 |
| 定价 | 按月订阅 | 按份计费 | 按份计费会抑制使用频率，订阅制鼓励多用，多用→多数据→更准确 |

这个案例的核心教训是：**垂直 Agent 的 PMF 不是"模型够强"，而是"工作流够深 + 数据飞轮转起来"**。ContractGuard 的技术壁垒不高（任何团队都能用 GPT-4 + RAG 做合同审查），但 6 个月积累的标注数据、Word 插件的工作流嵌入、以及律师社群的口碑，构成了后来者难以快速复制的壁垒。

### 19.2.5 垂直 Agent 的壁垒构建

进入一个垂直领域只是第一步，真正难的是建立壁垒。没有壁垒的 Agent 产品，竞争对手可以在几周内复制你的功能。以下是几种经过验证的壁垒构建策略：

**数据壁垒。** 持续积累领域特有的标注数据，让 Agent 的表现随时间提升。这些数据越积越多，后来者即使有同样的模型和算法，也无法在短期内追上你的数据优势。Harvey 在法律领域的合同标注数据、Khanmigo 在教育领域的学习行为数据，都是这种壁垒的体现。

**工作流壁垒。** 当 Agent 深度嵌入用户的工作流后，切换成本就变成了壁垒。用户不是在"用一个工具"，而是在"依赖一个流程"。替换 Agent 意味着重新配置整个工作流，这个成本高到大多数用户不愿意承担。

**合规壁垒。** 在医疗、金融、法律等强监管领域，合规认证本身就是壁垒。获得 HIPAA 认证、金融牌照、法律执业许可需要时间和资金投入，但一旦获得，竞争对手很难快速复制。

**网络壁垒。** 当你的 Agent 连接了足够多的上下游系统（CRM、ERP、邮件、日历），它就变成了一个"枢纽"——用户通过你的 Agent 访问和操作所有这些系统。替换你的 Agent，意味着重新建立所有这些连接。

最坚固的壁垒往往是几种壁垒的组合：数据壁垒让 Agent 越用越好，工作流壁垒让用户越用越离不开，合规壁垒让竞争对手越追越远。

## 19.3 Agent Marketplace 与生态

### 19.3.1 Agent 市场的形态

Agent Marketplace（Agent 市场）是 Agent 生态的核心基础设施。它连接了 Agent 的开发者和使用者，是 Agent 商业化的关键渠道。

目前 Agent 市场主要有三种形态：

**第一种：封闭市场（Walled Garden）。** 由单一公司运营，只允许符合其标准的 Agent 上架。GPT Store 是典型代表——只有基于 OpenAI 模型的 Agent 才能在上面发布。封闭市场的优势是质量可控、体验统一，劣势是创新受限、生态封闭。

**第二种：开放市场（Open Marketplace）。** 任何开发者都可以发布 Agent，平台提供基础设施和分发渠道。Coze、Dify 等平台正在向这个方向努力。开放市场的优势是生态繁荣、创新活跃，劣势是质量参差不齐、发现困难。

**第三种：协议网络（Protocol Network）。** 没有中心化的市场平台，而是通过统一的协议实现 Agent 之间的发现和交互。这类似于电子邮件的 SMTP 协议——你不需要在同一个平台上才能互相发邮件。Agent Protocol、Model Context Protocol（MCP）正在为这种形态奠定基础。

三种形态并非互斥，很可能会长期共存，就像 iOS App Store（封闭）、Google Play（半开放）和 Web 应用（协议网络）长期共存一样。

### 19.3.2 Agent 市场的核心机制

一个健康的 Agent 市场需要解决三个核心问题：质量、发现和信任。

**质量机制：** 如何确保市场上的 Agent 是安全可靠的？

- 事前审核：Agent 上架前通过自动化测试和安全审查
- 事后监控：运行时持续监控 Agent 的行为，发现异常立即下架
- 声誉系统：基于用户评价和使用数据构建 Agent 的信誉评分
- 保险机制：为高风险 Agent 提供责任保险，降低使用者的风险

**发现机制：** 用户如何找到适合自己的 Agent？

- 搜索：基于自然语言的 Agent 搜索（"帮我找一个能做财务分析的 Agent"）
- 推荐：基于用户画像和使用历史的个性化推荐
- 分类：按行业、功能、复杂度等维度的结构化分类
- 社交：好友推荐、社区讨论、KOL 测评

**信任机制：** 用户如何信任一个 Agent？

- 透明度：Agent 的决策过程可解释、可审计
- 沙箱执行：Agent 在受限环境中运行，不能访问未授权的资源
- 人类监督：高风险操作需要人类确认
- 可追溯：所有 Agent 的操作都有完整的日志记录

### 19.3.3 生态建设的飞轮

Agent 生态的繁荣依赖于一个飞轮效应：

```
开发者构建 Agent → 用户使用 Agent → 产生数据和反馈 → Agent 质量提升 → 吸引更多用户 → 吸引更多开发者
```

启动这个飞轮的关键是"先有鸡还是先有蛋"的问题——没有用户，开发者不愿意来；没有 Agent，用户也不愿意来。

解决这个冷启动问题的常见策略包括：

**种子开发者计划：** 用奖金、流量扶持等激励早期开发者入驻。Apple 在 App Store 早期就是这样做的。

**自建标杆 Agent：** 平台自己先做几个高质量的标杆 Agent，让用户体验到 Agent 的价值，同时为开发者提供参考。OpenAI 在 GPT Store 里内置了一些官方 GPT 就是这个思路。

**大客户锚定：** 先签下几个大企业客户，让他们提出需求，然后引导开发者来满足这些需求。这种方式尤其适合企业级 Agent 市场。

### 19.3.4 Agent 之间的生态关系

Agent 生态不仅有人与 Agent 的关系，还有 Agent 与 Agent 之间的关系。随着生态的成熟，Agent 之间的交互会越来越频繁和复杂。

几种典型的 Agent 间关系：

**编排关系：** 一个主 Agent 调度多个子 Agent 完成复杂任务。比如一个"商务旅行 Agent"可能需要调用"航班查询 Agent"、"酒店预订 Agent"和"日程管理 Agent"。

**协作关系：** 多个 Agent 对等协作，各司其职。比如"市场分析 Agent"和"财务分析 Agent"共同完成一份投资报告。

**竞争关系：** 同一任务的多个 Agent 竞争提供最优方案。用户可以同时调用多个 Agent，选择最好的结果。

**供应链关系：** 上游 Agent 的输出是下游 Agent 的输入。比如"数据采集 Agent"的输出是"数据分析 Agent"的输入，而"数据分析 Agent"的输出又是"报告生成 Agent"的输入。

这些关系的建立需要标准化的接口和协议，这就是 MCP（Model Context Protocol）和 A2A（Agent-to-Agent）协议的核心价值所在。

## 19.4 从 Demo 到 PMF

### 19.4.1 Demo 陷阱

在 Agent 领域，做一个令人惊叹的 Demo 从来没有这么容易过。几十行代码、一个精心设计的 Prompt，就能让 Agent 在演示中表现得令人印象深刻。

但 Demo 不等于产品。Demo 到 PMF（Product-Market Fit，产品-市场契合）之间的距离，可能是 Agent 创业中最容易被低估的鸿沟。

Demo 陷阱的典型表现：

**精心挑选的案例。** Demo 往往在精心挑选的场景下运行，但真实世界的需求是长尾的、模糊的、充满边缘情况的。你的法律 Agent 在审查标准合同时表现完美，但遇到非标准条款就可能完全失效。

**人为干预的流畅。** Demo 中的对话流程往往经过精心设计，但真实用户会问各种意想不到的问题、给各种不完整的输入、在各种意料之外的场景下使用。

**忽略成本的现实。** Demo 不关心推理成本，但产品必须关心。当用户量从十人变成十万人，你的 API 调用费用可能让你血本无归。

**缺失的工程化。** Demo 不需要考虑错误处理、日志监控、安全审计、性能优化，但产品必须考虑。这些"无聊"的工程工作可能占产品开发 80% 的工作量。

### 19.4.2 PMF 的信号

什么是 PMF？Marc Andreessen 的经典定义是：用户在你产品中发现了你从未想象过的用途；用户对产品有强烈的情感依赖；用户主动向他人推荐你的产品。

在 Agent 产品的语境下，PMF 的信号可以更具体地量化：

**留存率。** 周留存率超过 40% 是 PMF 的强信号。如果用户第一周用了，第二周不用了，说明你的 Agent 只是"有趣"而非"有用"。

**使用频率。** 目标用户平均每天使用超过 3 次，说明 Agent 已经融入了用户的日常工作流。

**付费意愿。** 当你提高价格时，用户不流失。这是最强但最难获得的 PMF 信号。

**自发传播。** 不依赖营销投放，用户自发推荐带来的新增占比超过 30%。

### 19.4.3 从 Demo 到 PMF 的五步法

基于对大量 Agent 产品的观察，我总结了一个从 Demo 到 PMF 的五步法：

**第一步：定义核心价值假设。**

你的 Agent 到底为用户解决什么问题？这个问题的答案必须是一个简洁的句子，而不是一段话。

好的核心价值假设："帮律师把合同审查时间从 3 小时缩短到 30 分钟。"

不好的核心价值假设："用 AI 帮助法律工作者提升工作效率。"——太模糊，无法验证。

**第二步：最小可用产品（MVP）。**

砍掉所有锦上添花的功能，只保留验证核心价值假设所需的最小功能集。MVP 不是一个缩水版的产品，而是一个精心设计的实验——它的唯一目的是用最低成本验证你的假设。

对于 Agent 产品来说，MVP 甚至可以不用 Agent——你可以用"人造 Agent"（Wizard of Oz）的方式，即用户以为在和 Agent 交互，实际上背后是人工在操作。这样可以零成本验证需求是否真实存在。

**第三步：深度用户研究。**

找到 10-20 个目标用户，让他们使用你的 MVP，然后深度访谈。不要问"你觉得这个产品怎么样"这种模糊的问题，而要问"你上次用它做了什么？过程中遇到了什么问题？如果没有它你会怎么做？"观察用户的实际行为，而不是听他们怎么说。

**第四步：快速迭代。**

基于用户反馈，以周为单位迭代产品。每次迭代聚焦于解决一个最关键的问题。不要试图一次解决所有问题——在找到 PMF 之前，速度比完美重要一万倍。

**第五步：寻找 PMF 信号。**

在迭代过程中，持续监控上面提到的 PMF 信号。当你观察到明确的 PMF 信号时，恭喜你，可以开始投入资源做规模化了。如果迭代了 10 次以上仍然没有 PMF 信号，可能需要考虑 Pivot（转型）。

### 19.4.4 Agent 产品的定价策略

定价是 Agent 商业化中最容易被忽视、但对商业成功影响最大的决策之一。

Agent 产品常见的定价模式：

**订阅制。** 月费或年费，提供一定额度的使用量。这是最常见也最被用户接受的定价模式。优势是收入可预测，劣势是难以应对用量差异大的用户。

**按次付费。** 每次任务执行收取一定费用。优势是公平（用多少付多少），劣势是用户对不确定性有焦虑——"这个任务到底会花多少钱？"

**效果付费。** 按照任务完成的效果收费，比如"帮你找到的线索转化为客户才收费"。优势是用户信任度高，劣势是收入不可预测且需要复杂的效果追踪机制。

**混合定价。** 基础订阅 + 超额按次付费。这可能是最平衡的方案——基础订阅覆盖用户的日常需求，超额按次付费覆盖高频率用户的额外使用。

定价的核心原则是：**让用户感知到的价值远大于他们支付的价格。** 对于 Agent 产品来说，最好的定价锚点是它替代的人工成本。如果你的 Agent 替代了一个月薪 2 万的员工的工作，那每月 2000 元的定价用户会觉得非常划算。

### 19.4.4.1 定价计算示例

理解定价模型最好的方式是亲手算一遍。以下对三种定价模式各给一个完整的计算示例。

**示例一：订阅制——代码审查 Agent "CodeGuard"**

假设 CodeGuard 是一个自动化代码审查 Agent，月费 199 元/用户。

```
单用户月成本估算：
  · 平均每用户每天审查 3 次 PR
  · 每次 PR 平均 200 行变更，约 2000 token 输入 + 1500 token 输出
  · 使用 GPT-4o-mini：输入 $0.15/M token，输出 $0.60/M token
  · 单次审查成本：(2000×0.15 + 1500×0.60) / 1,000,000 = $0.0012
  · 月审查次数：3 次/天 × 22 工作日 = 66 次
  · 月推理成本：66 × $0.0012 = $0.079 ≈ ¥0.57
  · 月基础设施成本（服务器、数据库）：¥5/用户
  · 月总成本：¥5.57

单位经济学：
  · 月收入：¥199
  · 月成本：¥5.57
  · 毛利率：(199 - 5.57) / 199 = 97.2%
  · 毛利：¥193.43/用户/月

  月活 1000 用户时：
  · 月收入：¥199,000
  · 月成本：¥5,570
  · 月毛利：¥193,430
```

订阅制在低边际成本场景下毛利率极高，但前提是用户确实高频使用。如果平均使用频率降至每周 1 次，毛利率仍达 95% 以上——订阅制的"不管用不用都收费"特性是收入稳定性的来源。

**示例二：按次付费——法律合同审查 Agent "LawCheck"**

LawCheck 按次收费，每次 ¥49。目标客户是中小企业法务部门。

```
单次任务成本估算：
  · 合同平均 15 页，约 8000 token 输入 + 3000 token 输出
  · 使用 GPT-4o：输入 $2.50/M token，输出 $10.00/M token
  · 单次推理成本：(8000×2.50 + 3000×10.00) / 1,000,000 = $0.05
  · 含 RAG 检索（法规库查询，额外 3000 token 输入）：+ $0.0075
  · 单次总推理成本：$0.0575 ≈ ¥0.42
  · 单次基础设施分摊：¥1.50（含向量数据库查询）
  · 单次总成本：¥1.92

单位经济学：
  · 单次收入：¥49
  · 单次成本：¥1.92
  · 毛利率：(49 - 1.92) / 49 = 96.1%
  · 单次毛利：¥47.08

  月 500 次调用时：
  · 月收入：¥24,500
  · 月成本：¥960
  · 月毛利：¥23,540
```

按次付费的毛利率同样很高，但收入波动大——法务审查有季节性（年底合同量大），淡季可能入不敷出。对创业团队来说，需要预留至少 3 个月的运营资金缓冲。

**示例三：效果付费——销售线索 Agent "LeadGen"**

LeadGen 按转化效果收费，每条转化为客户的线索收费 ¥500。

```
单条转化成本估算：
  · 平均需筛选 50 条原始线索才能产生 1 条转化
  · 每条原始线索分析成本：约 ¥0.80（LLM 调用 + 数据查询）
  · 50 条原始线索成本：50 × ¥0.80 = ¥40
  · 加上跟进沟通成本（3 次自动化触达 × ¥0.50/次）：¥1.50
  · 单条转化总成本：¥41.50

单位经济学：
  · 单条转化收入：¥500
  · 单条转化成本：¥41.50
  · 毛利率：(500 - 41.50) / 500 = 91.7%
  · 单条毛利：¥458.50

  月转化 100 条时：
  · 月收入：¥50,000
  · 月成本：¥4,150
  · 月毛利：¥45,850

  风险场景——转化率下降 50%：
  · 需筛选 100 条原始线索才能产生 1 条转化
  · 单条转化成本升至 ¥81.50
  · 毛利率降至 83.7%
  · 收入不变但成本翻倍，利润大幅缩水
```

效果付费的毛利率看似最高，但风险也最大——转化率受市场环境、竞争态势、客户预算等多重因素影响，波动不可控。建议设定最低转化率阈值，低于该阈值时与客户协商调整价格。

**关键洞察：** 三种定价模式的毛利率都可以达到 90% 以上，区别在于收入的可预测性和风险的承担方。订阅制收入最稳定，按次付费居中，效果付费最不确定。对于初创团队，建议从订阅制起步，用稳定的现金流支撑产品迭代，再逐步探索按次和效果付费模式。

### 19.4.5 一个真实的案例：从 Demo 到月入百万

让我们来看一个虚构但基于真实行业观察的案例——"法智通"，一个法律合同审查 Agent。

**Demo 阶段（第 1-2 周）：** 创始人用 GPT-4 API + 精心设计的 Prompt，做了一个能在 5 分钟内完成标准合同审查的 Demo。在路演中获得了律师们的一致好评。

**碰壁阶段（第 3-8 周）：** 上线 MVP 后，发现真实合同远比 Demo 复杂：非标准条款、多语言混合、行业特定条款……Agent 的准确率从 Demo 时的 90% 暴跌到 60%。用户试用后纷纷流失。

**深入阶段（第 9-16 周）：** 团队决定聚焦一个细分领域——劳动合同审查。深入学习了劳动法，收集了 5000+ 份标注过的劳动合同，用 RAG 增强了 Agent 的法律知识。劳动合同审查的准确率提升到 92%。

**PMF 阶段（第 17-24 周）：** 劳动合同 Agent 在 HR 社区获得口碑传播，3 个月内积累了 2000+ 付费用户。周留存率 55%，用户日均使用 2.3 次。找到了 PMF。

**增长阶段（第 25 周至今）：** 从劳动合同扩展到租赁合同、采购合同等相邻领域，付费用户突破 1 万，月收入超过 100 万元。

这个案例的关键启示：**不是去做一个"什么都能做"的 Agent，而是先在一个极窄的领域做到极致，然后逐步扩展。**

---

> **古典智慧点睛**：古人云"天时地利人和"——商业成功需要时机、赛道、团队的配合。Agent 的大时代是"天时"，垂直领域的深耕是"地利"，团队的执行力和场景理解力是"人和"。三者缺一，都难以走到 PMF 的彼岸。

## 习题

**习题 1：商业画布**

选择一个你感兴趣的垂直领域，完成一份 Agent 产品的精益画布（Lean Canvas）。包括：问题、解决方案、关键指标、独特优势、渠道、收入来源、成本结构、不容忽视的优势、客户群体。

**习题 2：PMF 验证设计**

为你的 Agent 产品设计一个 PMF 验证实验。包括：核心价值假设、MVP 的最小功能集、目标用户画像、关键指标定义、实验周期、判断标准。

**习题 3：定价模型推演**

针对你设计的 Agent 产品，推演三种不同的定价方案（订阅制、按次付费、效果付费），计算每种方案下的单位经济学（Unit Economics）：单用户收入、单用户成本、毛利率。找出最优定价策略。

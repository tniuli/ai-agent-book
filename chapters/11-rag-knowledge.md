## 11.1 RAG 基础

### 11.1.1 LLM 的知识困境

大语言模型的知识来自训练数据。这种"参数化知识"有三个根本性缺陷：

**时效性差**——训练数据有截止日期，模型无法获取训练之后的新信息。你问它昨天的新闻，它只会抱歉地说"我的知识截止于……"。

**可控性弱**——你无法精确控制模型"知道什么、不知道什么"。它可能对某个内部术语一无所知，却对某个公开八卦如数家珍。知识的边界是模糊的，而这种模糊在企业场景中是危险的。

**幻觉问题（Hallucination）**——当模型不确定时，它倾向于编造看似合理的答案，而非承认"我不知道"。这在专业领域尤其危险——一个编造的法律条文或药物剂量可能造成严重后果。更麻烦的是，模型的"自信"和"正确"并不相关——它编造的答案往往和真实答案一样信心满满。

RAG 的核心理念是：与其让模型"记住"所有知识，不如让模型"查找"所需知识。参数化知识擅长通用推理，检索知识擅长精确事实——两者互补，才是正道。

### 11.1.2 RAG 的基本流程

一个朴素 RAG（Naive RAG）的流程可以概括为三步：

```
朴素 RAG 流程：

  ┌──────────┐     ┌──────────┐     ┌──────────┐
  │  用户查询  │────→│  检索文档  │────→│  LLM生成  │
  │  Query   │     │ Retrieve │     │ Generate │
  └──────────┘     └──────────┘     └──────────┘
                        │
                        ↓
                  ┌──────────┐
                  │ 知识库/  │
                  │ 向量数据库│
                  └──────────┘

  1. 检索(Retrieve):  将查询转为向量，在知识库中找最相关的文档片段
  2. 增强(Augment):   将检索到的文档片段拼接到提示词中
  3. 生成(Generate):  LLM 基于增强后的上下文生成回答
```

用代码来表达这个流程：

```python
# version
def naive_rag(query: str, vector_store, llm) -> str:
    """朴素 RAG：检索 → 增强 → 生成"""
    # 1. 检索
    docs = vector_store.similarity_search(query, k=3)

    # 2. 增强
    context = "\n\n".join(doc.page_content for doc in docs)
    prompt = f"""基于以下参考资料回答问题。如果资料中没有相关信息，请说明。

参考资料：
{context}

问题：{query}
回答："""

    # 3. 生成
    response = llm.invoke(prompt)
    return response
```

这三步看似简单，但每一步都暗藏玄机：检索的精准度决定了上下文质量，增强的方式决定了 LLM 如何利用信息，生成的提示词设计决定了最终输出的准确性。

### 11.1.3 RAG vs 微调：何时用哪个？

很多人会问：RAG 和微调（Fine-tuning）有什么区别？什么时候用哪个？

```
RAG vs 微调 对比：

  ┌──────────────┬──────────────────────┬──────────────────────┐
  │    维度       │       RAG            │      微调             │
  ├──────────────┼──────────────────────┼──────────────────────┤
  │  知识来源     │ 外部知识库，实时更新    │ 训练数据，更新需重训   │
  │  适用场景     │ 事实性问答、文档查询    │ 风格适配、领域术语     │
  │  可解释性     │ 高（可追溯检索来源）    │ 低（知识融在参数中）   │
  │  成本         │ 检索成本 + 推理成本    │ 训练成本高，推理成本低 │
  │  幻觉控制     │ 较好（有检索约束）     │ 较差（依赖训练质量）   │
  │  实时性       │ 强（知识库即时更新）    │ 弱（需重新训练）       │
  └──────────────┴──────────────────────┴──────────────────────┘

  核心原则：
    - 知识会变 → RAG
    - 行为要改 → 微调
    - 两者都要 → 结合使用
```

打个比方：RAG 是"开卷考试"——你不用记住所有知识，只要知道去哪里查；微调是"上课培训"——通过反复练习内化知识和技能。开卷考试擅长精确事实，上课培训擅长方法论和风格。

一个常见的实践是：先用 RAG 注入领域知识，再通过少量微调调整模型的输出风格和术语习惯。这种"RAG + 微调"的组合在很多企业场景中效果最佳——模型既"懂行话"，又能"查资料"。

## 11.2 文档处理流水线

RAG 的效果，很大程度上取决于"喂"给它的知识质量。如果文档处理粗糙——切分位置不对、嵌入向量不准、索引结构混乱——再强大的 LLM 也只能从垃圾中产出垃圾。这里有两个尤其需要警惕的问题。一是切分粒度和位置的敏感性：如果在段落中间或表格行间切断，一个完整的论点会被拆成两个语义不完整的片段，检索时任何一个片段都不足以回答问题，所以应使用语义感知的切分策略（按段落、标题等自然边界切分而非固定长度）。二是"矛盾检索"：知识库中不同时期的文档可能包含相互矛盾的信息（如旧版政策 vs 新版政策），RAG 系统将两者同时返回给 LLM 时模型可能随意选择或产生混乱——应对方式是为每个片段附加时间戳和版本号元数据，检索后按时效性排序，并在提示词中明确要求 LLM 注意信息的时间线。

文档处理流水线是 RAG 的基础设施，就像酿酒前的选粮、洗粮、蒸煮——原料处理不好，再好的工艺也酿不出好酒。

### 11.2.1 流水线全景

```
文档处理流水线：

  ┌────────┐    ┌────────┐    ┌────────┐    ┌────────┐
  │  加载   │───→│  切分   │───→│  嵌入   │───→│  索引   │
  │ Load   │    │ Split  │    │ Embed  │    │ Index  │
  └────────┘    └────────┘    └────────┘    └────────┘
      │             │             │             │
  PDF/Word/     按段落/Token    文本→向量     向量数据库
  HTML/MD       递归切分       嵌入模型       Chroma/FAISS
```

每个环节都会影响最终效果，下面逐一讲解。

### 11.2.2 文档加载（Load）

文档加载的目标是将各种格式的文件统一为结构化的文档对象。LangChain 提供了丰富的 Loader：

```python
# version
"""
文档加载示例 —— 支持 PDF、Markdown、文本等多种格式
"""
from langchain_community.document_loaders import (
    PyPDFLoader,
    TextLoader,
    DirectoryLoader,
)

# 加载单个 PDF
pdf_loader = PyPDFLoader("docs/product_manual.pdf")
pdf_docs = pdf_loader.load()
# 每页一个 Document，metadata 包含 page 信息

# 加载单个文本文件
text_loader = TextLoader("docs/faq.txt", encoding="utf-8")
text_docs = text_loader.load()

# 批量加载目录下所有文件
dir_loader = DirectoryLoader(
    "docs/",
    glob="**/*.md",
    loader_cls=TextLoader,
    loader_kwargs={"encoding": "utf-8"},
)
all_docs = dir_loader.load()

print(f"加载了 {len(all_docs)} 个文档片段")
for doc in all_docs[:3]:
    print(f"  来源: {doc.metadata.get('source', 'unknown')}")
    print(f"  内容预览: {doc.page_content[:100]}...")
```

加载后的每个文档是一个 `Document` 对象，包含两部分：
- `page_content`：文本内容
- `metadata`：元数据（来源、页码等），后续检索时可用于过滤

文档加载看似简单，实则有不少坑：PDF 中的表格和图片可能丢失，编码不一致会导致乱码，超大文件需要分批加载。在实际项目中，这一步往往需要大量的清洗和预处理工作。

### 11.2.3 文档切分（Split）

文档切分是流水线中最影响效果的环节。切太大，检索不精确——一个 3000 字的片段可能涵盖了多个话题，检索时难以定位；切太小，丢失上下文——一句话脱离了上下文可能完全无法理解。

常见的切分策略：

**固定长度切分**——最简单，按字符数或 Token 数切分，但容易在句子中间断开，破坏语义完整性。

**递归字符切分（RecursiveCharacterTextSplitter）**——LangChain 的推荐方案。它的思路是"优先按大单位切，切不动再缩小单位"：先尝试按双换行（段落）切，段落太长则按单换行（行）切，行太长则按句子切，句子太长则按字符切。这样能尽量保持语义完整。

**语义切分**——用嵌入模型计算相邻文本的语义相似度，在相似度骤降处切分。效果最好，但成本最高，因为每段文本都需要调用嵌入模型。

```python
# version
"""
文档切分示例 —— 递归字符切分与元数据保留
"""
from langchain_text_splitters import RecursiveCharacterTextSplitter

splitter = RecursiveCharacterTextSplitter(
    chunk_size=500,       # 每个片段的最大字符数
    chunk_overlap=50,     # 相邻片段的重叠字符数
    separators=["\n\n", "\n", "。", "！", "？", "；", "，", " ", ""],
    length_function=len,
)

# 切分文档
chunks = splitter.split_documents(all_docs)

print(f"切分前: {len(all_docs)} 个文档")
print(f"切分后: {len(chunks)} 个片段")
for chunk in chunks[:3]:
    print(f"  长度: {len(chunk.page_content)} 字符")
    print(f"  元数据: {chunk.metadata}")
```

`chunk_overlap` 是一个关键参数。重叠部分确保切分处不会丢失上下文——就像裁缝拼接布料时预留的缝份，防止出现"信息断层"。通常设置为 `chunk_size` 的 10%-20%。

`separators` 参数对中文场景尤其重要。默认的英文分隔符以空格和句号为主，但中文文本的分隔逻辑不同——中文以句号"。"而非"."断句，以双换行分章节。上面的配置专门针对中文做了调整。

### 11.2.4 文本嵌入（Embed）

嵌入（Embedding）是将文本转换为高维向量的过程。语义相近的文本，向量距离也相近。这是向量检索的基础——就像给每段文本发了一个"语义身份证"，检索时通过比较身份证的相似度来找到相关内容。

```python
# version
"""
文本嵌入示例 —— 使用 OpenAI 嵌入模型
"""
from langchain_openai import OpenAIEmbeddings

embeddings = OpenAIEmbeddings(model="text-embedding-3-small")

# 嵌入单条文本
vector = embeddings.embed_query("什么是RAG？")
print(f"向量维度: {len(vector)}")  # 1536 维

# 嵌入文档列表
doc_vectors = embeddings.embed_documents(
    ["RAG是检索增强生成", "微调是模型训练方法", "向量数据库存储嵌入"]
)
print(f"嵌入了 {len(doc_vectors)} 条文档")
```

嵌入模型的选择直接影响检索质量。几个主流模型对比：

| 模型 | 维度 | 特点 |
|------|------|------|
| text-embedding-3-small | 1536 | 性价比高，通用场景首选 |
| text-embedding-3-large | 3072 | 精度更高，适合专业领域 |
| bge-large-zh | 1024 | 中文优化，开源可私有部署 |

如果你的场景以中文为主，bge-large-zh 是一个很好的选择——它在大规模中文语料上训练，对中文语义的理解远优于通用英文模型。而且它是开源的，可以私有部署，满足数据安全需求。

### 11.2.5 向量索引（Index）

索引是将嵌入向量存入向量数据库，供后续检索使用。向量数据库的核心操作是"近似最近邻搜索（Approximate Nearest Neighbor, ANN）"——在百万甚至亿级向量中，毫秒内找到与查询最相似的 Top-K 个。

```python
# version
"""
向量索引示例 —— 使用 Chroma 本地模式
"""
from langchain_community.vectorstores import Chroma
from langchain_openai import OpenAIEmbeddings

embeddings = OpenAIEmbeddings(model="text-embedding-3-small")

# 从文档片段创建向量库
vectorstore = Chroma.from_documents(
    documents=chunks,
    embedding=embeddings,
    persist_directory="./chroma_db",  # 持久化目录
)

# 相似度检索
results = vectorstore.similarity_search("如何部署Agent？", k=3)
for doc in results:
    print(f"来源: {doc.metadata.get('source', 'unknown')}")
    print(f"内容: {doc.page_content[:150]}...\n")
```

向量数据库的选择取决于场景：

| 数据库 | 特点 | 适用场景 |
|--------|------|---------|
| Chroma | 轻量，内置持久化 | 开发调试、小规模应用 |
| FAISS | Meta开源，性能极强 | 大规模批量检索 |
| Milvus | 分布式，支持GPU | 生产环境，亿级向量 |
| Qdrant | Rust实现，性能好 | 中大规模生产环境 |
| Weaviate | 内置混合检索 | 需要向量+关键词混合检索 |

起步阶段用 Chroma 或 FAISS 就够了，等数据量或并发量上来再迁移到 Milvus 或 Qdrant。

完整的 RAG 流水线代码见 `rag_pipeline.py`，它演示了从文档加载到向量检索的全流程，使用 Mock 嵌入模型，无需任何 API Key 即可运行。

## 11.3 混合检索

朴素 RAG 只用向量相似度检索，这在很多场景下是不够的。比如用户搜索"Python 3.12 的异常处理"，向量检索可能返回语义相关但版本不对的文档——它理解"异常处理"的含义，但不关心"3.12"这个精确版本号。关键词检索能精确定位"Python 3.12"，却无法理解"异常处理"和"错误捕获"是同义的。

混合检索（Hybrid Search）将两种检索方式结合，取长补短。

### 11.3.1 为什么需要混合检索？

```
检索方式对比：

  ┌─────────────────────────────────────────────────────────────┐
  │  用户查询："部署失败 error code 503"                          │
  │                                                             │
  │  向量检索结果：                                              │
  │    ✓ "服务启动异常的排查步骤"  （语义相关，但可能不涉及503）    │
  │    ✓ "系统崩溃的诊断方法"      （语义相关，但太宽泛）          │
  │    ✗ 精确包含"503"的文档      （语义距离可能较远）            │
  │                                                             │
  │  关键词检索结果（BM25）：                                    │
  │    ✓ "error code 503 处理方案" （精确匹配503）                │
  │    ✗ "503公交路线"             （关键词匹配但语义无关）        │
  │    ✗ "服务异常排查"            （语义相关但无503关键词）       │
  │                                                             │
  │  混合检索结果：                                              │
  │    ✓ "error code 503 处理方案" （关键词命中 + 语义相关）       │
  │    ✓ "服务异常的排查步骤"       （语义补充）                   │
  │    ✓ "503 相关的系统日志分析"   （关键词补充）                 │
  └─────────────────────────────────────────────────────────────┘
```

向量检索是"模糊的聪明"——它理解语义，但可能漏掉精确的关键词。关键词检索（BM25 是最经典的算法）是"精确的笨"——它严格匹配字面，但无法理解同义表达。混合检索让两者互相补位。

### 11.3.2 结果融合：倒数秩融合

两个检索通道各返回一个排序列表，如何合并？最常用的方法是倒数秩融合（Reciprocal Rank Fusion, RRF）。它的公式简洁而有效：

```
RRF_score(doc) = Σ 1 / (k + rank_i)

其中：
  rank_i 是文档在第 i 个排序列表中的排名（从0开始）
  k 是平滑常数（通常取60），防止排名靠前的文档过度主导
```

RRF 的巧妙之处在于：它不依赖原始分数，只依赖排名。这意味着向量检索的余弦相似度和 BM25 的 TF-IDF 分数不需要归一化到同一尺度，直接用排名融合即可。

```python
# version
"""
混合检索实现 —— 向量检索 + BM25 关键词检索 + 倒数秩融合
"""
from langchain_community.vectorstores import Chroma
from langchain_community.retrievers import BM25Retriever
from langchain_openai import OpenAIEmbeddings
from langchain_core.documents import Document


def reciprocal_rank_fusion(
    doc_lists: list[list[Document]],
    k: int = 60,
) -> list[Document]:
    """倒数秩融合（RRF）

    将多个排序列表融合为一个排序，公式：
    RRF_score(doc) = Σ 1 / (k + rank_i)

    k 值越大，排名靠前的文档优势越小，排序越平滑。
    """
    fused_scores: dict[str, float] = {}
    doc_map: dict[str, Document] = {}

    for docs in doc_lists:
        for rank, doc in enumerate(docs):
            content_hash = hash(doc.page_content)
            if content_hash not in doc_map:
                doc_map[content_hash] = doc
            fused_scores[content_hash] = (
                fused_scores.get(content_hash, 0) + 1.0 / (k + rank + 1)
            )

    # 按融合分数降序排列
    sorted_items = sorted(fused_scores.items(), key=lambda x: x[1], reverse=True)
    return [doc_map[h] for h, _ in sorted_items]


def hybrid_search(
    query: str,
    vectorstore: Chroma,
    bm25_retriever: BM25Retriever,
    top_k: int = 5,
) -> list[Document]:
    """混合检索：向量检索 + BM25 + RRF 融合"""
    # 向量检索
    vector_docs = vectorstore.similarity_search(query, k=top_k)

    # BM25 关键词检索
    keyword_docs = bm25_retriever.invoke(query)

    # 倒数秩融合
    fused_docs = reciprocal_rank_fusion([vector_docs, keyword_docs])

    return fused_docs[:top_k]
```

### 11.3.3 重排序（Reranking）

混合检索融合了两个检索通道的结果，但融合排序未必是最优的——它只是"两份榜单的折中"。重排序模型（Reranker）专门做一件事：对候选文档与查询的相关性进行精细打分，重新排列。

```
混合检索 + 重排序流程：

  ┌──────────┐
  │ 用户查询  │
  └────┬─────┘
       │
  ┌────┴────┐
  │         │
  ↓         ↓
向量检索   关键词检索
(BM25/分词)
  │         │
  └────┬────┘
       ↓
  RRF 融合排序
       │
       ↓
  ┌──────────┐
  │ 重排序    │  ← Cross-Encoder 精细打分
  │ Reranker │     对每个(query, doc)对独立评分
  └────┬─────┘
       ↓
  最终排序结果
```

为什么重排序能提升效果？因为向量检索用的是 Bi-Encoder——查询和文档分别独立编码为向量，然后计算向量距离。这种方式快，但"看不到"查询和文档之间的交互。重排序用的 Cross-Encoder 同时编码查询和文档，能捕捉两者之间的细粒度交互——比如文档中的"503"是否确实是在讨论错误码，而不是公交路线。

代价是速度：Cross-Encoder 需要对每个候选文档单独推理，所以只对少量候选文档重排序（通常 10-50 个），而非全库重排。这就是"先粗筛，后精排"的两阶段策略。

## 11.4 高级 RAG

朴素 RAG 有一个隐含的假设：用户的查询本身就是好的检索词。但现实远非如此——用户的问题可能模糊、口语化，甚至包含误解。高级 RAG 的核心思想是：在检索前后增加"智能环节"，让检索更精准，让生成更可靠。

"学而不思则罔，思而不学则殆"——光检索不推理（不判断相关性）是"罔"，光推理不检索（不查证事实）是"殆"。高级 RAG 就是让 Agent 既"学"又"思"。

### 11.4.1 查询改写（Query Rewriting）

用户说"这玩意儿怎么报错这么多"，Agent 需要理解"这玩意儿"指什么，"报错"是什么类型的错误。查询改写让 LLM 先理解用户意图，再生成更适合检索的查询。

查询改写有多种策略：

**查询扩展（Query Expansion）**：将一个查询扩展为多个不同角度的查询，增加检索的召回率。比如"部署失败"可以扩展为"部署报错"、"服务启动失败"、"应用发布异常"等。

**查询分解（Query Decomposition）**：将复杂查询拆分为多个子查询。比如"公司差旅政策中国际和国内有什么区别"可以拆分为"国内差旅政策"和"国际差旅政策"两个子查询。

**假设文档嵌入（HyDE）**：先让 LLM 生成一个"假设性答案"，再用这个假设答案（而非原始查询）去检索。假设答案的语义更接近真实文档，能提高检索的精准度。

```python
# version
"""
查询改写示例 —— 将口语化查询转为结构化检索词
"""
from langchain_core.prompts import ChatPromptTemplate
from langchain_openai import ChatOpenAI

rewrite_prompt = ChatPromptTemplate.from_messages([
    ("system", """你是一个查询改写专家。将用户的口语化问题改写为更适合检索的查询。

要求：
1. 保留原始意图
2. 补充隐含的关键词
3. 使用更精确的技术术语
4. 生成1-3个不同角度的改写查询

输出格式（每行一个查询）：
1. 改写查询1
2. 改写查询2
3. 改写查询3"""),
    ("user", "原始问题：{query}"),
])

llm = ChatOpenAI(model="gpt-4o-mini", temperature=0)

def rewrite_query(query: str) -> list[str]:
    """改写查询，返回多个检索友好的查询"""
    chain = rewrite_prompt | llm
    result = chain.invoke({"query": query})

    queries = [query]  # 保留原始查询
    for line in result.content.split("\n"):
        line = line.strip().lstrip("0123456789.-) ")
        if line and len(line) > 3:
            queries.append(line)

    return queries[:4]  # 最多4个查询


# 示例
queries = rewrite_query("这玩意儿怎么报错这么多")
for q in queries:
    print(f"  → {q}")
# 输出示例：
#   → 这玩意儿怎么报错这么多
#   → 程序运行报错频繁的排查方法
#   → 应用错误日志分析与常见错误码
#   → 服务异常错误率高的问题定位
```

查询改写的成本是一次额外的 LLM 调用，但收益是显著的——更好的检索词意味着更少需要检索的轮次，整体上反而可能降低成本。

### 11.4.2 自适应检索（Adaptive Retrieval）

不是每个问题都需要检索。问"1+1等于几"去查文档毫无意义，问"公司最新的差旅政策"不查就是瞎编。自适应检索让 Agent 自己判断是否需要检索，以及检索什么。

```
自适应检索决策流程：

  ┌──────────┐
  │ 用户查询  │
  └────┬─────┘
       ↓
  ┌──────────┐     需要检索     ┌──────────┐
  │ 检索判断  │──────────────→ │ 执行检索  │
  │ Need RAG?│                │ RAG      │
  └────┬─────┘                └────┬─────┘
       │ 不需要                     │
       ↓                            ↓
  ┌──────────┐                ┌──────────┐
  │ 直接回答  │                │ 基于检索  │
  │ LLM Only │                │ 增强回答  │
  └──────────┘                └──────────┘
```

判断逻辑可以基于规则（关键词匹配、问题分类），也可以用 LLM 做意图判断。后者更灵活，但有额外的 LLM 调用成本。一个实用的折中方案是：先用规则做快速分类（如检测到特定领域关键词则触发检索），对规则无法判断的模糊问题再用 LLM 决策。

### 11.4.3 递归检索（Recursive Retrieval）

有些问题需要多次检索才能找到答案。比如"张三负责的项目中，哪个预算超支了？"——第一次检索"张三负责的项目"，第二次检索"这些项目的预算情况"。递归检索让 Agent 像人类查资料一样，先找到线索，再沿着线索深挖。

```python
# version
"""
递归检索示例 —— 多轮检索逐步聚焦
"""
from langchain_core.prompts import ChatPromptTemplate
from langchain_openai import ChatOpenAI


def recursive_retrieve(
    query: str,
    vectorstore,
    llm: ChatOpenAI,
    max_rounds: int = 3,
) -> list[str]:
    """递归检索：根据前一轮检索结果生成下一轮查询"""
    collected_context: list[str] = []
    current_query = query

    decompose_prompt = ChatPromptTemplate.from_messages([
        ("system", """基于已有检索结果，判断是否需要进一步检索。
如果需要，生成下一个检索查询。
如果信息已足够，输出"DONE"。

原始问题：{original_query}
已检索到的信息：{collected}
当前检索结果：{current_result}"""),
        ("user", "请判断并生成下一步检索查询："),
    ])

    chain = decompose_prompt | llm

    for i in range(max_rounds):
        # 检索
        docs = vectorstore.similarity_search(current_query, k=3)
        result_text = "\n".join(doc.page_content for doc in docs)
        collected_context.append(result_text)

        # 判断是否继续
        decision = chain.invoke({
            "original_query": query,
            "collected": "\n---\n".join(collected_context),
            "current_result": result_text,
        })

        next_query = decision.content.strip()
        if next_query == "DONE" or not next_query:
            break
        current_query = next_query

    return collected_context
```

递归检索的关键风险是"检索雪崩"——每一轮检索可能触发更多的子检索，导致指数级增长。必须设置最大检索轮次（如3轮）和总文档数上限（如10个片段），同时让 LLM 做好"信息充分性判断"——如果已经足够回答问题，及时停止。

### 11.4.4 朴素 RAG vs 高级 RAG 对比

```
┌──────────────┬──────────────────────────┬──────────────────────────┐
│    维度       │     朴素 RAG             │      高级 RAG            │
├──────────────┼──────────────────────────┼──────────────────────────┤
│  查询处理     │ 原始查询直接检索           │ 查询改写/分解/扩展        │
│  检索策略     │ 单一向量检索              │ 混合检索 + 重排序         │
│  检索决策     │ 每次都检索                │ 自适应判断是否需要检索     │
│  检索深度     │ 单轮检索                  │ 递归检索，多轮深挖        │
│  生成约束     │ 弱约束，可能忽略检索结果    │ 强约束，优先使用检索内容   │
│  适用场景     │ 简单问答、Demo 验证        │ 生产环境、专业领域        │
│  实现复杂度   │ 低                       │ 中-高                    │
│  成本         │ 低                       │ 中（额外的 LLM 调用）     │
└──────────────┴──────────────────────────┴──────────────────────────┘

关键区别：高级 RAG 在检索前后增加了"智能层"——
  检索前：查询理解、改写、路由
  检索后：重排序、过滤、压缩
  检索中：自适应决策、递归深挖
```

我的建议是：**从朴素 RAG 开始，逐步增加高级特性**。先把基础流水线跑通，观察到哪些环节是瓶颈，再针对性地增加查询改写、混合检索、重排序等。不要一上来就搞"全副武装"的高级 RAG——过度工程化不仅增加维护成本，还可能引入新的问题。

## 11.5 Agent + RAG

前面我们讲的 RAG 都是"被动"的——用户问一个问题，系统检索一次，生成一个答案。但在 Agent 场景中，RAG 应该是一种"主动"能力——Agent 根据任务需要，自主决定何时检索、检索什么、检索结果是否足够。

### 11.5.1 从 RAG Pipeline 到 RAG Tool

把 RAG 从"流水线"变成"工具"，是 Agent + RAG 的关键转变：

```
RAG 的两种范式：

  Pipeline 模式（被动）：
  ┌────────┐     ┌────────┐     ┌────────┐
  │  查询   │────→│  检索   │────→│  生成   │
  └────────┘     └────────┘     └────────┘
  每次请求都检索，检索逻辑硬编码

  Agent 模式（主动）：
  ┌────────┐     ┌────────────────────────┐
  │  查询   │────→│       Agent            │
  └────────┘     │  ┌──────┐  ┌──────┐   │
                 │  │ 思考  │→│ 决策  │   │
                 │  └──────┘  └──┬───┘   │
                 │               │       │
                 │     ┌─────────┼────┐  │
                 │     ↓         ↓    ↓  │
                 │  ┌──────┐ ┌─────┐ ... │
                 │  │ 检索 │ │计算 │     │
                 │  │ 工具 │ │工具 │     │
                 │  └──────┘ └─────┘     │
                 └────────────────────────┘
  Agent 自主决定是否检索、检索什么
```

Agent 模式的核心优势是**灵活性**：
- 简单问题直接回答，不浪费检索资源
- 复杂问题可以多次检索，逐步聚焦
- 检索结果不相关时，Agent 可以换一种方式重试
- 可以结合其他工具（搜索、计算、API）协同完成任务

这种灵活性在复杂任务中尤为重要。想象一个场景：用户问"我们公司的差旅政策和竞品公司相比有什么优势？"Agent 需要：先检索公司差旅政策，再搜索竞品信息，然后综合比较——这种多步推理和工具组合是 Pipeline 模式无法实现的。

### 11.5.2 将 RAG 封装为 Agent 工具

```python
# version
"""
将 RAG 封装为 LangChain 工具 —— Agent 可自主调用
"""
from langchain_core.tools import tool
from langchain_community.vectorstores import Chroma
from langchain_openai import OpenAIEmbeddings


# 初始化向量库（假设已创建）
embeddings = OpenAIEmbeddings(model="text-embedding-3-small")
vectorstore = Chroma(
    persist_directory="./chroma_db",
    embedding_function=embeddings,
)


@tool
def knowledge_search(query: str) -> str:
    """在企业知识库中检索相关信息。

    当你需要查找公司内部文档、技术规范、流程说明等
    知识库中的信息时使用此工具。

    Args:
        query: 检索查询，应使用精确的关键词和术语
    """
    docs = vectorstore.similarity_search(query, k=3)
    if not docs:
        return "未在知识库中找到相关信息。"

    results = []
    for i, doc in enumerate(docs, 1):
        source = doc.metadata.get("source", "未知来源")
        results.append(f"[{i}] 来源: {source}\n{doc.page_content}")

    return "\n\n---\n\n".join(results)


@tool
def knowledge_search_by_type(query: str, source_type: str = "") -> str:
    """在企业知识库中按类型过滤检索。

    当你需要限定文档类型（如只查技术文档、只查流程规范）时使用。

    Args:
        query: 检索查询
        source_type: 文档类型过滤，如 "tech", "process", "faq"
    """
    filter_dict = {}
    if source_type:
        filter_dict = {"source_type": source_type}

    docs = vectorstore.similarity_search(
        query, k=3, filter=filter_dict
    )
    if not docs:
        return "未在知识库中找到符合条件的文档。"

    results = []
    for i, doc in enumerate(docs, 1):
        results.append(f"[{i}] {doc.page_content}")

    return "\n\n---\n\n".join(results)
```

工具的描述（description）非常重要——它就是 LLM 决定"何时调用这个工具"的唯一依据。描述要清晰说明工具的功能、适用场景和参数含义。模糊的描述会导致 LLM 在不需要检索时乱检索，需要检索时又不检索。

## 11.6 知识图谱增强

向量检索的本质是"语义相似度"——它擅长找到"说了类似内容"的文档，但不擅长回答需要跨文档推理的问题。比如"公司所有项目中，哪些用到了微服务架构且团队规模超过10人？"——这需要关联"项目"→"技术架构"→"团队规模"等多跳信息，纯向量检索很难搞定。

知识图谱（Knowledge Graph）天然适合表达实体间的关系。GraphRAG 将知识图谱与向量检索结合，让 RAG 既能"模糊匹配"，又能"精确推理"。

### 11.6.1 GraphRAG 的核心思路

```
GraphRAG vs 向量 RAG：

  向量 RAG（基于片段）：
  问题 → 嵌入 → 在文档片段中找相似的 → 拼接上下文 → 生成
  擅长："X是什么" / "怎么做X"
  不擅长："X和Y有什么关系" / "所有X中满足条件Y的有哪些"

  GraphRAG（基于实体和关系）：
  问题 → 识别实体/关系 → 在图谱中遍历 → 聚合社区信息 → 生成
  擅长："X和Y有什么关系" / 多跳推理
  不擅长：简单事实查询（杀鸡用牛刀）

  ┌──────────────────────────────────────────────────────────┐
  │  知识图谱示例：                                           │
  │                                                          │
  │   [项目A] ──使用──→ [微服务]    [项目A] ──团队──→ [团队1]  │
  │     │                   ↑              │                  │
  │    负责人              也使用           规模: 15人         │
  │     ↓                   │              ↓                  │
  │   [张三]          [项目B] ──团队──→ [团队2]               │
  │                          规模: 8人                        │
  │                                                          │
  │  查询："哪些项目用微服务且团队超10人？"                      │
  │  向量RAG：可能检索到项目A和项目B的文档片段，但难以关联       │
  │  GraphRAG：直接在图谱中遍历 → 项目A满足，项目B不满足        │
  └──────────────────────────────────────────────────────────┘
```

GraphRAG 最初由微软研究院在2024年提出，核心创新是用 LLM 从文档中自动抽取实体和关系，构建知识图谱，再通过社区检测和摘要来回答全局性问题。

### 11.6.2 GraphRAG 的构建流程

GraphRAG 的构建分为两阶段：

**阶段一：索引（Indexing）**

1. 文档切分（与向量 RAG 相同）
2. 实体与关系抽取：用 LLM 从文本中提取实体（人、组织、技术等）和关系（使用、负责、属于等）
3. 构建知识图谱：将实体作为节点，关系作为边
4. 社区检测：用图算法（如 Leiden 算法）将图谱划分为社区——紧密关联的实体群
5. 社区摘要：用 LLM 为每个社区生成摘要描述

**阶段二：查询（Querying）**

1. 局部查询（Local Search）：针对具体实体，在图谱中获取其邻域信息，适合"X是什么"类问题
2. 全局查询（Global Search）：将查询映射到社区，聚合相关社区的摘要，适合"整体趋势/全局概览"类问题
3. 混合查询：结合向量检索和图谱遍历

```
GraphRAG 索引流程：

  ┌────────┐    ┌────────────┐    ┌────────────┐
  │  文档   │───→│ 实体关系抽取 │───→│  知识图谱   │
  │  Docs  │    │ LLM抽取    │    │  Graph    │
  └────────┘    └────────────┘    └─────┬──────┘
                                       │
                                ┌──────┴───────┐
                                ↓              ↓
                          ┌──────────┐   ┌──────────┐
                          │ 社区检测  │   │ 向量索引  │
                          │ Leiden  │   │ Embedding│
                          └────┬─────┘   └──────────┘
                               ↓
                          ┌──────────┐
                          │ 社区摘要  │
                          │ LLM生成  │
                          └──────────┘

  GraphRAG 查询流程：

  ┌──────────┐     ┌────────────────────┐     ┌──────────┐
  │  用户查询  │────→│  局部/全局/混合路由  │────→│  LLM生成  │
  └──────────┘     └────────────────────┘     └──────────┘
                        │           │
                  局部查询       全局查询
                        │           │
                        ↓           ↓
                   图谱邻域遍历    社区摘要聚合
```

GraphRAG 最大的成本在于索引阶段——每个文档片段都需要 LLM 调用来抽取实体和关系。对于 10 万篇文档，索引成本可能达到数百美元（GPT-4 级别模型）。此外，知识图谱需要随文档更新而重建，增量更新不如向量库方便。

### 11.6.3 三种 RAG 范式对比

```
┌──────────────┬────────────────────┬────────────────────┬────────────────────┐
│    维度       │    朴素/高级 RAG    │     GraphRAG       │   混合 GraphRAG     │
├──────────────┼────────────────────┼────────────────────┼────────────────────┤
│  知识表示     │ 文档片段            │ 实体 + 关系图       │ 片段 + 图谱         │
│  检索方式     │ 向量相似度          │ 图遍历 + 社区聚合   │ 向量 + 图遍历       │
│  擅长问题     │ "X是什么/怎么做"    │ "X和Y什么关系"      │ 通用               │
│  多跳推理     │ 弱                 │ 强                  │ 强                 │
│  全局性问题   │ 弱（片段碎片化）    │ 强（社区摘要）      │ 强                 │
│  构建成本     │ 低                 │ 高（LLM抽取）       │ 高                 │
│  更新成本     │ 低（增量添加片段）  │ 高（需重建图谱）    │ 中                 │
│  适用场景     │ 事实性问答          │ 关系推理、全局分析  │ 企业级综合应用      │
│  典型框架     │ LangChain + Chroma │ Microsoft GraphRAG │ LightRAG           │
└──────────────┴────────────────────┴────────────────────┴────────────────────┘

选型建议：
  - 80% 的企业知识库场景 → 高级 RAG（向量 + 关键词 + 重排序）就够了
  - 需要多跳推理（如合规审计、供应链分析）→ GraphRAG
  - 既有事实问答又有关系推理 → 混合 GraphRAG
  - 从简单开始，按需升级——不要一上来就上 GraphRAG
```

"学而不思则罔"——GraphRAG 的价值在于它能做向量检索做不到的"推理"，但如果你的问题压根不需要推理，上 GraphRAG 就是"罔"——费力不讨好。

### 11.7.3 关键设计决策

在动手写代码之前，先理清几个关键设计决策：

**为什么用 Mock 数据？**——RAG 系统的核心逻辑是检索+生成，与具体的嵌入模型无关。Mock 数据让读者无需配置 API Key 就能运行完整代码，理解核心流程。读者可以把 Mock 数据替换为真实文档，把 Mock 嵌入替换为 OpenAI 嵌入，系统其他部分无需改动。

**为什么用混合检索？**——企业知识库的文档通常包含大量专业术语和编号（如"Secret"、"error code 503"），关键词检索能精确定位这些精确标识，而向量检索能理解语义关联。混合检索在企业场景中几乎是标配。

**为什么让 Agent 自主决策？**——企业知识库的用户问题五花八门：有些只需要简单事实查询，有些需要跨文档推理，有些根本与知识库无关。硬编码的"每次都检索"既浪费资源，又可能引入噪音。让 Agent 自主判断，是更优雅的方案。

### 11.7.4 核心代码解读

完整的可运行代码见 `enterprise_kb_agent.py`。这里重点解读几个核心设计：

**混合检索引擎**

```python
def hybrid_search(query, vectorstore, bm25_retriever, top_k=5):
    """混合检索：向量检索 + BM25 + RRF 融合"""
    vector_docs = vectorstore.similarity_search(query, k=top_k)
    keyword_docs = bm25_retriever.invoke(query)
    fused_docs = reciprocal_rank_fusion([vector_docs, keyword_docs])
    return fused_docs[:top_k]
```

三行代码，却完成了"双通道检索+融合排序"的全部工作。向量检索负责语义匹配，BM25 负责关键词匹配，RRF 负责融合排序。这种"粗筛+融合"的架构可以轻松扩展——如果未来想加入第三个检索通道（如知识图谱检索），只需将新通道的结果加入 `doc_lists` 即可。

**RAG 工具定义**

```python
@tool
def knowledge_search(query: str) -> str:
    """在企业知识库中检索相关信息。"""
    docs = hybrid_search(query, _vectorstore, _bm25_retriever, top_k=3)
    ...
```

将检索封装为工具，是 Agent + RAG 的核心模式。工具的描述（docstring）至关重要——它是 LLM 决定"何时调用这个工具"的唯一依据。描述要清晰说明功能和适用场景。

**Agent 图结构**

Agent 使用 LangGraph 构建，核心是一个"判断→检索/回答"的循环。当 LLM 认为需要检索时，它发出工具调用；当检索结果返回后，LLM 再决定是直接回答还是继续检索。这种"自主决策"模式比硬编码的 Pipeline 灵活得多。

### 11.7.6 进阶改造方向

这个实战项目是一个起点，以下是几个有价值的改造方向：

**1. 增加重排序**——在混合检索后加入 Cross-Encoder 重排序，提升检索精度。可以试试 `langchain_cohere` 的 `CohereRerank` 或 `bge-reranker`。

**2. 增加查询改写**——在检索前让 LLM 改写查询，提升召回率。特别适合口语化的用户输入。

**3. 增加对话记忆**——当前 Agent 是无状态的。用 LangGraph 的检查点机制增加对话历史，让 Agent 能处理追问和上下文相关的问题。

**4. 增加文档更新机制**——当知识库中的文档更新时，如何增量更新向量索引？Chroma 支持 `add_documents` 和 `delete`，可以实现增量更新。

**5. 增加评估指标**——如何量化 RAG 的效果？可以参考 RAGAS 框架，评估检索的精确率、召回率和生成的忠实度。

## 📌 Prompt Engineering 融入：RAG 提示词——如何引导 Agent 优先使用检索结果

RAG 的效果不仅取决于检索质量，还取决于 LLM 是否"听话"——它是否真的基于检索结果回答，而不是自作主张地编造。提示词在这里扮演了"指挥官"的角色。

### 核心原则

```
RAG 提示词优先级：

  1. 检索结果（最高优先级）
     "基于以下参考资料回答问题"

  2. 用户指令（次高优先级）
     "请用简洁的语言总结"

  3. 模型参数化知识（最低优先级）
     模型自己的训练知识，仅作为补充
```

### RAG 系统提示词模板

```
你是"{agent_name}"，一个企业知识助手。

## 核心规则
1. 优先使用检索结果回答问题。如果检索结果包含相关信息，必须基于检索结果回答。
2. 如果检索结果不包含相关信息，明确说明"知识库中暂无相关内容"，不要编造。
3. 引用检索结果时，标注来源文档，如"根据《XX规范》..."。
4. 如果用户的问题不需要检索（如常识问题），可以直接回答。

## 回答格式
📋 回答：[基于检索结果的回答]
📄 来源：[引用的文档来源]
💡 补充：[如有必要，补充模型知识]

## 检索策略
- 首次检索未找到相关信息时，可以尝试：
  - 换一种方式描述查询
  - 使用更通用的关键词
  - 使用 knowledge_search_by_type 按类型检索
- 最多检索2次，避免过度检索。
```

### RAG 用户提示词模板

在将检索结果拼接到提示词时，格式也很关键：

```
请基于以下参考资料回答问题。

## 参考资料
{retrieved_documents}

## 问题
{user_query}

## 回答要求
1. 回答必须基于参考资料中的信息
2. 如果参考资料不足以回答问题，请说明缺少什么信息
3. 引用信息时标注来源
```

### 常见问题与对策

| 问题 | 原因 | 对策 |
|------|------|------|
| LLM 忽略检索结果 | 提示词约束不够强 | "必须基于检索结果回答"，加"必须" |
| LLM 编造未检索到的信息 | 模型"自作主张" | "如果检索结果中没有，说明没有，不要编造" |
| LLM 照搬检索原文 | 缺乏"组织"指令 | "用自己的话综合整理检索结果" |
| 检索结果矛盾 | 不同文档说法不一 | "如果检索结果有矛盾，指出矛盾并给出各来源" |
| LLM 过度检索 | 每次都调用检索工具 | "判断是否需要检索，简单问题直接回答" |

**一个实用的技巧**：在系统提示词中加入"角色约束"——"你是一个严谨的企业助手，你的每一个回答都可能影响同事的决策。因此，你只回答有据可查的内容，绝不编造。"这种"责任化"的约束比简单的"不要编造"更有效。

## 开放讨论

1. **RAG 的检索结果一定是"真相"吗？** 如果知识库本身包含过时或错误的信息，RAG 只是让 LLM 更"自信地犯错"。如何在 RAG 系统中引入"知识可信度评估"——让 Agent 不仅检索内容，还评估内容本身的可靠性？

2. **GraphRAG 的构建成本值得吗？** 微软 GraphRAG 的索引阶段需要大量的 LLM 调用来抽取实体和关系，对于百万级文档，索引成本可能上千美元。在什么场景下，这种成本投入是值得的？有没有更轻量的替代方案？

3. **Agent 的"自主检索"真的比"每次都检索"好吗？** 自适应检索引入了一个新问题：Agent 可能判断失误——该检索时不检索，不该检索时乱检索。这种"决策错误"比"每次都检索"的浪费更危险吗？如何在灵活性和安全性之间找到平衡？

# 第2章 Agent 核心架构

> 知己知彼，百战不殆。——《孙子兵法·谋攻篇》

上一章我们认识了 AI Agent 的全貌——它不再是那个只会"一问一答"的聊天机器人，而是一个能够自主感知环境、做出决策、采取行动的智能体。那么，一个 Agent 到底由哪些部件组成？这些部件如何协作？为什么有的 Agent 简单高效，有的却层层嵌套、如同一支军队？本章将从最底层的"感知-决策-行动"模型讲起，逐步拆解 Agent 的核心架构，带你理解 ReAct 循环的工作机制、四大组件（LLM、工具、记忆、规划）的职责与协作方式，区分单 Agent 与多 Agent 架构的适用场景，并了解反射型、规划型、协作型三种架构模式。你会发现，看似复杂的 Agent 系统，其实都遵循着几条简洁而深刻的设计原则。

***

## 2.1 感知-决策-行动模型

### 2.1.1 从生命体到智能体

如果我问你：一个最简单的生命体需要什么？答案无非三件事——**感知**周围环境、**判断**如何应对、**执行**相应动作。草履虫趋利避害如此，人类日常生活亦如此。你听到闹钟响（感知），决定再赖五分钟（决策），翻身继续睡（行动）——这就是一个完整的感知-决策-行动（Perceive-Decide-Act，简称 PDA）循环。

AI Agent 的运行逻辑，本质上就是对这一生命基本模式的工程化实现：

```
┌─────────────────────────────────────────────────────────┐
│              感知-决策-行动模型（PDA Loop）                │
│                                                         │
│   ┌──────────┐      ┌──────────┐      ┌──────────┐      │
│   │          │      │          │      │          │      │
│   │  感知     │─────▶│  决策    │─────▶│  行动     │      │
│   │ Perceive │      │ Decide   │      │   Act    │     │
│   │          │      │          │      │          │     │
│   └──────────┘      └──────────┘      └──────────┘     │
│        ▲                                    │           │
│        │                                    │           │
│        └──────────── 环境反馈 ◄──────────────┘           │
│                                                         │
└─────────────────────────────────────────────────────────┘
```

这个模型虽然简单，却是所有 Agent 架构的基石。无论你看到多么复杂的 Agent 系统，把它剥到最底层，一定是一个 PDA 循环在不断运转。

### 2.1.2 三个阶段详解

**感知（Perceive）**——Agent 的"感官"

感知阶段的核心任务是**信息采集与表征**。Agent 需要从环境中获取两类信息：

- 外部信息：用户的指令、API 返回的数据、网页的内容、传感器的读数……一切来自外部世界的信息。
- 内部信息：自身的状态（已完成了哪些步骤、当前目标是什么）、记忆中存储的历史经验。

感知并不是简单地"拿到数据"就完事，更关键的是**信息的选择与编码**。一个优秀的 Agent 应当知道在什么情境下关注什么信息，就像一个经验丰富的棋手不会去看棋盘边上的花纹，而是聚焦于棋子的位置关系。

```
感知阶段的输入/输出：

输入：环境状态 + 用户指令 + 内部记忆
         │
         ▼
   ┌─────────────┐
   │ 信息筛选     │  ← 哪些信息与当前任务相关？
   │ 格式编码     │  ← 如何将信息转化为 LLM 可理解的表示？
   │ 上下文组装   │  ← 如何组织信息使决策更加准确？
   └─────────────┘
         │
         ▼
输出：结构化的上下文（Context）
```

**决策（Decide）**——Agent 的"大脑"

决策阶段是整个循环的核心，通常由大语言模型（LLM）承担。LLM 接收感知阶段输出的结构化上下文，基于自身的推理能力，决定下一步该做什么。

这里有一个关键点：**决策不等于生成一段文本**。在 Agent 的语境下，决策的输出通常是一个结构化的"行动指令"——调用哪个工具、传递什么参数、是否需要继续推理、是否可以给出最终答案。

```
决策阶段的输入/输出：

输入：结构化上下文 + 可用工具列表 + 行动历史
         │
         ▼
   ┌─────────────┐
   │ 推理分析     │  ← 当前情况如何？目标是什么？
   │ 选项评估     │  ← 有哪些可用行动？各有什么利弊？
   │ 行动选择     │  ← 选择最优行动
   └─────────────┘
         │
         ▼
输出：行动指令（Action）
```

**行动（Act）**——Agent 的"手脚"

行动阶段负责将决策阶段的输出付诸实施。这可能是一次工具调用、一次数据库查询、一条消息的发送，或者一个物理动作的执行。行动阶段的执行结果会成为下一轮感知的输入，形成闭环。

```
行动阶段的输入/输出：

输入：行动指令
         │
         ▼
   ┌─────────────┐
   │ 参数校验     │  ← 参数是否合法？权限是否足够？
   │ 工具执行     │  ← 调用具体的工具/API
   │ 结果捕获     │  ← 获取执行结果与可能的异常
   └─────────────┘
         │
         ▼
输出：执行结果（Observation）
```

### 2.1.3 闭环的力量

PDA 模型之所以强大，不在于三个阶段的独立运作，而在于**闭环**——行动的结果反馈到感知，驱动新一轮的决策。这种闭环使得 Agent 能够：

1. 应对不确定性：第一轮行动的结果可能出乎意料，闭环让 Agent 有机会调整策略。
2. 处理复杂任务：一个复杂任务往往需要多轮"感知→决策→行动"才能完成，闭环提供了持续推演的能力。
3. 自我纠错：如果行动失败，错误信息会通过感知阶段传回，Agent 可以尝试其他方案。

## 2.2 ReAct 循环

### 2.2.1 思考与行动的张力

在设计 Agent 时，一个根本性的问题始终存在：**何时思考，何时行动？**

一种极端是"先想清楚再动手"——让 Agent 在行动之前进行详尽的推理，把所有可能性都想一遍，再选最优方案。这听起来很完美，但现实中往往行不通：第一，很多信息只有行动之后才能获得（比如搜索结果）；第二，过度思考容易陷入"分析瘫痪"（Analysis Paralysis），在复杂问题面前反复权衡却迟迟不行动。

另一种极端是"先干再说"——让 Agent 一有想法就立刻行动，走一步看一步。这种方式反应快，但容易盲目，可能走很多弯路甚至走错方向。

2022 年，姚期智团队的 YAO Shunyu 等人提出了 ReAct（Reasoning + Acting）范式，优雅地解决了这个张力——**不是在思考和行动之间二选一，而是让它们交替进行、相互促进**。

### 2.2.2 ReAct 的核心思想

ReAct 的核心思想可以用一句话概括：**在每一步中，Agent 先进行推理（Thought），再采取行动（Action），然后观察结果（Observation），如此循环**。

```
┌──────────────────────────────────────────────────────────────┐
│                    ReAct 循环                                 │
│                                                              │
│   ┌──────────┐     ┌──────────┐     ┌───────────┐           │
│   │          │     │          │     │           │           │
│   │ Thought  │────▶│ Action   │────▶│Observation│           │
│   │ 推理思考  │     │ 执行行动  │     │ 观察结果   │           │
│   │          │     │          │     │           │           │
│   └──────────┘     └──────────┘     └───────────┘           │
│        ▲                                   │                 │
│        │           进入下一轮               │                 │
│        └───────────────────────────────────┘                 │
│                                                              │
│   最终输出：Answer（最终答案）                                  │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

假设用户问："LangChain 的最新版本是什么？"

```
第1轮：
  Thought: 我需要搜索 LangChain 的最新版本信息
  Action:  Search["LangChain latest version"]
  Observation: LangChain v0.3.0 was released in October 2024...

第2轮：
  Thought: 搜索结果提到了 v0.3.0，但这是2024年的信息，
           我需要确认是否有更新版本
  Action:  Search["LangChain version 2025 2026"]
  Observation: LangChain v0.3.2 is the latest stable release...

第3轮：
  Thought: 我现在有了足够的信息来回答用户的问题
  Answer:  LangChain 的最新稳定版本是 v0.3.2
```

在每一轮中，Agent 都是先思考（当前需要做什么），再行动（执行具体操作），然后观察（获取反馈）。推理指导行动，行动的结果又反过来丰富推理——两者形成了一个螺旋上升的过程。

### 2.2.3 ReAct vs 纯推理 vs 纯行动

把 ReAct 和另外两种范式做一个对比：

| 维度   | 纯推理（Chain-of-Thought） | 纯行动（Act-only） | ReAct    |
| ---- | --------------------- | ------------- | -------- |
| 核心思路 | 只思考，不行动               | 只行动，不思考       | 思考与行动交替  |
| 信息来源 | 仅靠已有知识                | 仅靠外部查询        | 两者兼备     |
| 错误处理 | 难以发现自身错误              | 容易走错方向        | 通过观察及时纠错 |
| 可解释性 | 推理过程清晰但可能脱离实际         | 行动序列清晰但缺乏理由   | 推理和行动都可见 |
| 适用场景 | 封闭式推理题                | 简单查询任务        | 大多数实际场景  |

纯推理就像一个闭门造车的学者，逻辑严密但可能脱离现实；纯行动就像一个埋头苦干的工匠，效率高但可能方向偏了。ReAct 则是两者的结合——**既抬头看路，又低头拉车**。

### 2.2.4 ReAct 的伪代码实现

下面是一个简化的 ReAct 循环伪代码，帮助理解其核心逻辑：

```python
def react_loop(agent, question, max_steps=10):
    """ReAct 循环的核心实现"""
    context = f"问题: {question}\n"

    for step in range(max_steps):
        # 1. LLM 根据上下文生成 Thought 和 Action
        response = agent.llm.generate(context)

        # 2. 解析出 Thought（推理过程）
        thought = parse_thought(response)
        context += f"Thought: {thought}\n"

        # 3. 解析出 Action（行动指令）
        action = parse_action(response)

        # 4. 判断是否已经得出最终答案
        if action.type == "Finish":
            return action.answer

        # 5. 执行行动，获取观察结果
        observation = agent.execute_tool(action.name, action.args)
        context += f"Action: {action}\nObservation: {observation}\n"

    return "未能完成任务：已达到最大步数限制"
```

这段伪代码虽然简化了大量工程细节，但清晰地展示了 ReAct 的核心流程：**Thought → Action → Observation 的循环，直到产出最终答案**。

> 古语点睛：**"知行合一"**——王阳明。ReAct 循环正是"知行合一"的工程实现：知（Thought）指导行（Action），行（Observation）反哺知（下一轮 Thought），知与行不再是割裂的两件事，而是一个不可分割的整体。

## 2.3 Agent 的四大组件

如果说 PDA 模型描述了 Agent "怎么运转"，那么四大组件描述的就是 Agent "由什么构成"。一个完整的 Agent，离不开四个核心组件：大语言模型（LLM）、工具（Tools）、记忆（Memory）、规划（Planning）。这里需要特别提醒的是，很多初学者容易把 Agent 等同于"LLM + 提示词"，认为只要模型够强、提示词够好就能搞定一切——但事实是，没有工具，Agent 就是"纸上谈兵"；没有记忆，Agent 就是"金鱼脑"；没有规划，Agent 就是"无头苍蝇"。四个组件之间存在明显的短板效应，缺一不可。因此在设计 Agent 时，建议先评估四个组件各自的成熟度，优先补齐最弱的那一环。

```
┌──────────────────────────────────────────────────────────────────┐
│                     Agent 四大组件关系图                          │

│                        ┌─────────┐                               │

│                    ┌──▶│         │──┐                            │

│                    │        │       │                            │
│                    │   协调  │    指导 │                            │
│                    │   调度  │    监督 │                            │
│                    │        ▼       │                            │
│              ┌─────┴────────────┐   │                            │
│              │                  │   │                            │
│              │     LLM (将)     │   │                            │
│              │   大语言模型      │   │                            │
│              │  推理与决策核心    │   │                            │
│              │                  │   │                            │
│              └──┬──────────┬───┘   │                            │
│                 │          │        │                            │
```

### 2.3.1 LLM——Agent 之"将"

在《孙子兵法》的"五事"（道、天、地、将、法）中，"将"是军队的指挥官，负责审时度势、运筹帷幄。在 Agent 的四组件中，LLM 扮演的正是"将"的角色——它是整个 Agent 的推理与决策核心。

LLM 在 Agent 中的职责包括：

- 理解与推理：理解用户意图、分析当前情境、推导下一步行动。
- 工具选择：根据当前需要，从可用工具列表中选择最合适的工具。
- 结果解读：解读工具返回的结果，判断是否需要继续行动。
- 答案生成：在适当的时候，综合所有信息，生成最终回答。

选择 LLM 的关键考量：

| 考量维度  | 说明                                 |
| ----- | ---------------------------------- |
| 推理能力  | Agent 需要强推理能力来处理复杂任务，通常需要更强的模型     |
| 工具调用  | 是否原生支持 function calling / tool use |
| 上下文窗口 | 长任务需要更大的上下文窗口来容纳历史信息               |
| 延迟与成本 | Agent 可能需要多轮循环，延迟和成本会累积            |
| 指令遵循  | 能否严格按照 prompt 格式输出（如 JSON 格式的行动指令） |

需要注意的是，**并非所有场景都需要最强大的 LLM**。对于简单的查询类 Agent，一个小参数量的模型可能就够用了；而对于需要复杂推理的场景，才需要调用最顶尖的模型。这也是为什么很多 Agent 框架支持"模型路由"——根据任务复杂度动态选择不同能力的模型。

### 2.3.2 工具——Agent 之"法"

"法"是军队的组织编制、军纪法规，是将领意志的延伸。工具之于 Agent，正如法之于军——**LLM 的能力再强，没有工具就只能"纸上谈兵"；有了工具，LLM 的智慧才能真正"落地"**。

工具的本质是**对外部世界的接口**。通过工具，Agent 能够：

- 获取信息：搜索引擎、数据库查询、API 调用
- 操作外部系统：发送邮件、创建文件、提交代码
- 执行计算：代码解释器、数学计算引擎
- 感知环境：浏览器自动化、传感器读取

一个好的工具设计需要考虑：

```
工具设计的四要素：

1. 功能明确 ── 一个工具只做一件事，做好一件事
                避免创建"万能工具"，这会让 LLM 在选择时产生困惑

2. 描述清晰 ── 工具的名称和描述要准确无歧义
                LLM 靠文字理解工具用途，描述模糊 = 选择错误

3. 参数规范 ── 每个参数的类型、范围、默认值都要明确定义
                越严格的约束，LLM 犯错的空间越小

4. 错误友好 ── 执行失败时返回有意义的错误信息
                好的错误信息能帮助 LLM 自我纠错
```

工具数量的权衡也是一个重要课题。给 Agent 太少的工具，它可能无法完成任务；给太多工具，LLM 会在选择时犹豫不决，甚至选错。实践经验表明，**5-15 个精心设计的工具**往往比 50 个粗粒度的工具效果更好。

### 2.3.3 记忆——Agent 之"地"

"地"是战场的地形地貌，是远近距离、险阻平坦的代名词。没有对地形的熟悉，再好的将帅也无法用兵。记忆之于 Agent，正如地形之于战争——**它是 Agent 对"过去"的了解，是决策的基础和依据**。

Agent 的记忆通常分为三个层次：

```
┌────────────────────────────────────────────────────────┐
│              Agent 记忆的三个层次                        │
│                                                        │
│  ┌──────────────────────────────────────────────┐      │
│  │          长期记忆（Long-term Memory）          │      │
│  │                                              │      │
│  │  · 用户偏好、历史对话摘要                      │      │
│  │  · 知识库、文档索引                            │      │
│  │  · 持久化存储（向量数据库、关系型数据库）        │      │
│  │  · 跨会话保留                                  │      │
│  └──────────────────────────────────────────────┘      │
│           ▲                           ▲                 │
│           │ 写入                      │ 检索            │
│           │                           │                 │
│  ┌──────────────────────────────────────────────┐      │
│  │          工作记忆（Working Memory）            │      │
│  │                                              │      │
│  │  · 当前任务的 Scratchpad（草稿板）             │      │
│  │  · 中间推理结果、待办事项                      │      │
│  │  · 存在于 LLM 的上下文窗口中                   │      │
│  │  · 随任务结束而消失                            │      │
│  └──────────────────────────────────────────────┘      │
│           ▲                           ▲                 │
│           │ 写入                      │ 读取            │
│           │                           │                 │
│  ┌──────────────────────────────────────────────┐      │
│  │          短期记忆（Short-term Memory）         │      │
│  │                                              │      │
│  │  · 当前对话的上下文                            │      │
│  │  · 最近几轮的 Thought/Action/Observation       │      │
│  │  · 受上下文窗口大小限制                        │      │
│  │  · 单次会话内有效                              │      │
│  └──────────────────────────────────────────────┘      │
│                                                        │
└────────────────────────────────────────────────────────┘
```

**短期记忆**是最基本的记忆形式，就是当前对话中的历史消息。在 ReAct 循环中，每一轮的 Thought、Action、Observation 都会被追加到短期记忆中，作为下一轮推理的输入。短期记忆的容量受限于 LLM 的上下文窗口——当历史信息超过窗口大小时，就需要进行截断或摘要。

**工作记忆**是一个更精细的概念，它指的是 Agent 在执行任务过程中主动维护的"草稿板"。比如，一个 Agent 在做研究报告时，可能会在工作记忆中维护一个"已查找的信息"列表和"还需要查找的问题"列表。工作记忆的引入，使得 Agent 能够更好地组织自己的思维过程，避免在海量信息中迷失。

**长期记忆**使得 Agent 能够跨会话积累知识和经验。最常见的实现方式是向量数据库（Vector Database）：将信息编码为向量并存储，检索时通过语义相似度匹配找到相关内容。长期记忆让 Agent 从"金鱼记忆"进化为"博学多闻"。

记忆系统的核心挑战是**检索**——如何在海量信息中找到当前最相关的内容？这涉及到向量检索、重排序（Reranking）、记忆压缩等技术，我们将在后续章节详细探讨。

### 2.3.4 规划——Agent 之"道"

"道"是五事之首，是战争的正义性、民众的认同、共同的信念。它是最高层级的指引，决定了"为什么而战"。规划之于 Agent，正如"道"之于战争——**它是任务的最高策略，决定了 Agent 的目标分解、步骤编排和资源调配**。

没有规划的 Agent 就像一个没有作战方案的军队，虽然每个士兵都很勇猛（LLM 能力强），工具也很精良，但缺乏方向和协调，最终只会事倍功半。

常见的规划策略包括：

1. 任务分解（Task Decomposition）

将一个复杂任务拆解为多个子任务，逐一完成。最经典的实现是"思维树"（Tree of Thoughts）——将任务可能的解决路径展开为一棵树，通过搜索策略（如广度优先、深度优先）找到最优路径。

```
原始任务：写一篇关于 AI Agent 的调研报告
                │
                ├── 子任务1：调研 Agent 的定义与发展历史
                │       ├── 查找学术论文
                │       └── 查找技术博客
                ├── 子任务2：调研主流 Agent 框架
                │       ├── 分析 LangChain Agent
                │       ├── 分析 AutoGen
                │       └── 分析 CrewAI
                ├── 子任务3：分析技术挑战与未来方向
                │       ├── 整理当前技术瓶颈
                │       └── 预测未来趋势
                └── 子任务4：整合报告
                        ├── 撰写各章节
                        └── 校对与排版
```

2. 自我反思（Self-Reflection）

Agent 在执行过程中对自己的行动进行审视和纠正。比如 ReAct 中的 Observation 环节就是一种隐式的反思——观察到行动结果不理想，就会调整策略。更高级的反思机制如 Reflexion，会让 Agent 显式地"复盘"自己的失败经历，生成改进建议，并在下一次尝试中应用。

3. 动态重规划（Dynamic Replanning）

计划不是一成不变的。当行动结果与预期不符时，Agent 需要能够调整计划。这就像战场上的形势瞬息万变，将帅必须根据实际情况灵活调整战术。动态重规划要求 Agent 具备"元认知"能力——不仅知道自己在做什么，还能评估当前计划是否仍然可行，并在必要时推倒重来。

```python
# 规划组件的简化伪代码
class Planner:
    def plan(self, task):
        """生成初始计划"""
        subtasks = self.decompose(task)
        return Plan(subtasks)

    def replan(self, original_plan, execution_result):
        """根据执行结果动态调整计划"""
        if execution_result.success:
            return original_plan  # 继续原计划
        else:
            # 分析失败原因，调整后续计划
            adjusted = self.adjust(original_plan, execution_result)
            return adjusted

    def reflect(self, full_trace):
        """对完整的执行轨迹进行反思"""
        lessons = self.analyze_failures(full_trace)
        return lessons  # 供未来的规划参考
```

### 2.3.5 四组件的协作关系

四个组件不是孤立存在的，它们之间有着紧密的协作关系：

- 规划指引全局，决定 LLM 应该在什么阶段做什么事；
- LLM 是执行核心，调用工具、读写记忆、生成推理；
- 工具是 LLM 的手脚，将推理转化为实际行动；
- 记忆是 LLM 的后盾，提供决策所需的信息和经验。

用一句话总结：**规划定方向，LLM 做决策，工具出行动，记忆供依据**。

> 古语点睛：《孙子兵法》"五事"——道、天、地、将、法。Agent 四组件与此暗合：**规划为道**（指引方向），**LLM 为将**（运筹决策），**记忆为地**（提供地利），**工具为法**（规范行动）。五事缺一不可，四组件同样如此。

## 2.4 单 Agent vs 多 Agent 架构

### 2.4.1 什么时候一个 Agent 就够了？

很多任务用一个 Agent 就能完成。当你只需要一种角色、一种视角、一套工具时，单 Agent 架构是最简单、最高效的选择。

单 Agent 架构的优势：

- 简单可控：只有一个决策主体，行为路径清晰可追踪
- 开发成本低：只需设计一套 prompt 和工具集
- 调试容易：出了问题，排查范围有限
- 延迟更低：不需要跨 Agent 通信

典型的单 Agent 适用场景：信息查询助手、代码补全工具、简单的客服机器人、个人日程管理……这类任务目标明确、步骤有限、工具需求单一，一个 Agent 完全可以胜任。

### 2.4.2 什么时候需要多个 Agent？

当任务变得复杂，一个 Agent 开始力不从心时，多 Agent 架构就派上用场了。以下几种情况通常需要引入多 Agent：

角色分工：不同任务需要不同专业能力。比如软件开发中，产品经理、架构师、程序员、测试工程师各司其职，很难让一个 Agent 同时扮演好所有角色。

视角多样性：有些问题需要从不同角度审视。比如一场辩论，正方和反方各有一个 Agent，由裁判 Agent 评判，能产出更全面的分析。

安全隔离：某些操作需要权限控制。比如一个 Agent 负责审核内容，另一个 Agent 负责执行发布，两者分离可以防止单点决策失误导致严重后果。

并行效率：多个独立子任务可以由不同 Agent 并行执行，大幅缩短总耗时。

### 2.4.3 多 Agent 的常见编排模式

```

│                                                                 │

│  Agent1 ──▶ Agent2 ──▶ Agent3 ──▶ 最终输出                     │
│  (调研)     (撰写)     (审核)                                    │
│                                                                 │
│  模式3：辩论模式（Debate）                                       │
│       ┌─── Agent_A（正方）───┐                                  │
│  用户 ─┤                      ├─▶ Agent_Judge（裁判）─▶ 结论    │
│       └─── Agent_B（反方）───┘                                  │
│                                                                 │
│  模式4：对等协作模式（Peer Collaboration）                       │

│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

主从模式是最常见的多 Agent 架构。一个主 Agent（Orchestrator）负责任务分解和分配，多个工作 Agent 负责具体执行。主 Agent 掌握全局视野，工作 Agent 专注各自领域。这种模式的好处是结构清晰、容易扩展；缺点是主 Agent 成为单点，如果它的规划出错了，整个团队都会跑偏。

流水线模式将任务分解为一系列顺序步骤，每个 Agent 负责其中一个步骤，前一个 Agent 的输出是后一个 Agent 的输入。就像工厂流水线一样，每个工人只做自己的工序。这种模式适合流程固定、步骤明确的任务，但灵活性较差——如果中间某个环节出了问题，整个流水线都需要停顿。

辩论模式让多个 Agent 从不同立场出发，对同一问题进行讨论和辩论，由裁判 Agent 做出最终判断。这种模式特别适合需要多角度分析的问题，能有效减少单一视角的偏见。

对等协作模式中，Agent 之间没有主从关系，它们共享一个公共的上下文空间，通过消息传递进行协商和协作。这种模式最灵活，但也最难控制——如何避免 Agent 之间的循环讨论、如何确保收敛到一个结论，都是需要仔细设计的问题。

一个实用的经验法则：**能用单 Agent 解决的，不要用多 Agent**。多 Agent 架构带来的通信开销、协调复杂度和调试难度，往往超出预期。只有在单 Agent 确实无法胜任时，才引入多 Agent 架构。

## 2.5 架构模式分类

根据 Agent 的智能程度和架构复杂度，我们可以将其分为三种基本模式：反射型、规划型和协作型。这三种模式不是互斥的，而是一种递进关系——反射型是最基础的形态，规划型在其上增加了前瞻能力，协作型则引入了多 Agent 的团队协作。

### 2.5.1 反射型架构（Reactive Agent）

反射型 Agent 是最简单的 Agent 形态，它的核心特点是\*\*"刺激-响应"\*\*——感知到什么，就做出对应的反应，不进行复杂的推理和规划。

```
┌────────────────────────────────────────────────────────┐
│              反射型架构（Reactive Agent）                 │
│                                                        │
│   输入 ──────▶ [规则匹配 / LLM 单次推理] ──────▶ 输出   │
│                                                        │
│   特点：                                                │
│   · 无状态或极有限的状态                                 │
│   · 无前瞻规划，只看当前输入                             │
│   · 响应速度快                                          │
│   · 行为可预测                                          │
│                                                        │
│   示例：                                                │
│   · 简单的意图分类 + 槽位填充机器人                      │
│   · 基于规则的自动回复系统                               │
│   · 单轮工具调用（如"帮我查天气"→调用天气API）           │
│                                                        │
└────────────────────────────────────────────────────────┘
```

反射型架构的优点是简单、快速、可控；缺点是缺乏"大局观"，无法处理需要多步推理的复杂任务。它就像一个条件反射极快的武者——对手出一拳，立刻格挡——但如果对手设了一个连环计，就很难应对了。

在现代 Agent 系统中，纯粹的反射型架构已经比较少见，但"反射"作为一种**底层机制**仍然广泛存在——即使是最复杂的规划型 Agent，在具体的工具调用环节也往往是反射式的。

### 2.5.2 规划型架构（Planning Agent）

规划型 Agent 在反射型的基础上，增加了**前瞻和回溯**的能力。它不仅考虑"当前这一步该做什么"，还会规划"未来几步要怎么走"，并在执行过程中根据反馈调整计划。

```
┌──────────────────────────────────────────────────────────────┐
│              规划型架构（Planning Agent）                       │
│                                                              │
│   ┌──────────┐    ┌──────────┐    ┌──────────────────┐       │
│   │          │    │          │    │                  │       │
│   │  规划器   │───▶│  执行器   │───▶│  反思器          │       │
│   │ Planner  │    │Executor  │    │Reflector        │       │
│   │          │    │          │    │                  │       │
│   └──────────┘    └──────────┘    └──────────────────┘       │
│        ▲                                │                     │
│        │         反馈与调整               │                     │
│        └────────────────────────────────┘                     │
│                                                              │
│   特点：                                                      │
│   · 有明确的规划阶段，生成步骤序列                              │
│   · 执行过程中可动态调整计划                                   │
│   · 具备自我反思能力                                          │
│   · 能处理多步骤的复杂任务                                     │
│                                                              │
│   示例：                                                      │
│   · ReAct 循环的 Agent                                       │
│   · Plan-and-Solve Agent                                     │
│   · Reflexion Agent                                          │
│   · 代码生成与调试 Agent                                      │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

规划型架构的核心是"规划-执行-反思"的三段式循环，这与我们在 2.2 节讨论的 ReAct 一脉相承，但更进一步：

- ReAct 是**逐步规划**（每步只规划一个行动），而规划型架构可以在开始时就**生成完整计划**，也可以在执行中动态调整。
- 规划型架构引入了**反思机制**，不仅能从行动结果中学习，还能显式地分析"为什么失败"并提炼经验。

一个更直观的例子：

```
任务："帮我对比三家云服务商的 GPU 实例价格"

规划阶段：
  Step 1: 搜索 AWS GPU 实例定价
  Step 2: 搜索 GCP GPU 实例定价
  Step 3: 搜索 Azure GPU 实例定价
  Step 4: 整理为对比表格
  Step 5: 分析性价比并给出推荐

执行阶段：
  Step 1: [执行搜索] → 获得AWS定价信息 ✓
  Step 2: [执行搜索] → 获得GCP定价信息 ✓
  Step 3: [执行搜索] → 页面无法访问 ✗
  ── 触发反思：Azure 官网可能改版了，换个搜索关键词
  Step 3': [重新搜索] → 获得 Azure 定价信息 ✓
  Step 4: [生成表格] → 对比表生成完成 ✓
  Step 5: [分析推荐] → 输出分析报告 ✓
```

看到 Step 3 的处理了吗？规划型 Agent 在执行失败后不是直接放弃，而是反思失败原因、调整策略后重新尝试。这种"遇挫不馁、灵活应变"的能力，正是规划型架构的核心价值。

### 2.5.3 协作型架构（Collaborative Agent）

协作型架构是 Agent 架构的最高形态，它引入了**多 Agent 协作**。多个 Agent 组成团队，各自承担不同角色，通过通信机制协调行动，共同完成复杂任务。

```
┌──────────────────────────────────────────────────────────────────┐
│              协作型架构（Collaborative Agent）                      │
│                                                                  │
│                      ┌──────────────┐                            │
│                      │  编排 Agent   │                            │
│                      │ (Orchestrator)│                            │
│                      └──────┬───────┘                            │
│                             │                                    │
│              ┌──────────────┼──────────────┐                    │
│              │              │              │                     │
│              ▼              ▼              ▼                     │
│      ┌──────────┐   ┌──────────┐   ┌──────────┐               │
│      │ 调研Agent │   │ 撰写Agent │   │ 审核Agent │               │
│      │          │   │          │   │          │               │
│      │ · 搜索   │   │ · 整理   │   │ · 校验   │               │
│      │ · 采集   │   │ · 撰写   │   │ · 打回   │               │
│      │ · 摘要   │   │ · 排版   │   │ · 批准   │               │
│      └──────────┘   └──────────┘   └──────────┘               │
│           │              │              │                        │
│           └────── 共享记忆空间 ──────────┘                       │
│                                                                  │
│   特点：                                                          │
│   · 多个 Agent 分工协作                                           │
│   · 通过消息传递或共享记忆协调                                     │
│   · 每个Agent可有独立的LLM、工具集和记忆                           │
│   · 系统整体能力超越任何单个Agent                                  │
│                                                                  │
│   示例：                                                          │
│   · AutoGen 多Agent对话                                           │
│   · CrewAI 角色团队                                               │
│   · MetaGPT 软件开发团队                                          │
│   · ChatDev 虚拟软件公司                                          │
│                                                                  │
└──────────────────────────────────────────────────────────────────┘
```

协作型架构的关键设计决策包括：

1. 通信机制：Agent 之间如何交换信息？直接的消息传递（点对点）、共享的黑板系统（Blackboard）、还是通过中间人（如 Orchestrator）转发？不同的通信机制在效率、灵活性和可调试性上有不同的权衡。

2. 协调策略：谁决定哪个 Agent 在什么时候做什么？集中式（由编排 Agent 统一调度）还是分布式（Agent 自主协商）？集中式更容易控制，但编排 Agent 是瓶颈；分布式更灵活，但可能出现死锁或循环。

3. 共享状态：Agent 之间共享多少信息？完全共享（所有 Agent 看到相同上下文）还是按需共享（只传递必要信息）？完全共享简单但上下文窗口压力大；按需共享高效但信息可能不完整。

4. 冲突解决：当多个 Agent 意见不一致时怎么办？投票、权重、权威仲裁？冲突解决机制直接影响系统的鲁棒性。

### 2.5.4 三种架构模式的对比

```
┌─────────────────────────────────────────────────────────────────┐
│               三种架构模式对比                                    │
│                                                                 │
│  维度        反射型          规划型           协作型             │
│  ────────  ──────────    ──────────     ──────────            │
│  复杂度     ★☆☆          ★★☆           ★★★                  │
│  智能程度   ★☆☆          ★★☆           ★★★                  │
│  前瞻能力   无             有              有                   │
│  反思能力   无             有              有                   │
│  多步推理   无             有              有                   │
│  团队协作   无             无              有                   │
│  开发成本   低             中              高                   │
│  调试难度   低             中              高                   │
│  适用场景   简单响应       复杂任务        团队级任务            │
│                                                                 │
│  典型框架   LlamaIndex     LangChain      AutoGen              │
│             简单Chain      Agent          CrewAI               │
│                            Reflexion      MetaGPT              │
│                                                                 │
└─────────────────────────────────────────────────────────────────┘
```

需要强调的是，这三种模式**不是非此即彼的选择**，而是一个渐进增强的谱系：

- 在协作型架构中，每个 Agent 内部通常采用规划型架构；
- 在规划型架构中，具体的工具调用环节通常是反射式的；
- 一个系统可以在不同层次使用不同的架构模式。

这种"嵌套组合"的思路，就像现实中的组织架构——公司是一个协作型组织，每个部门内部有规划流程，而具体员工执行任务时又是条件反射式的。

> 古语点睛：**"凡事预则立，不预则废"**——《礼记·中庸》。从反射型到规划型再到协作型，核心的进化方向就是"预"——预见更多步骤、预备更多方案、协调更多力量。但切记，"预"不是越复杂越好，而是**恰到好处**。

### 2.5.5 架构选择量化决策矩阵

上面的决策框架给出了定性的判断逻辑，但在实际选型时，我们需要更量化的评估方法。下面是一个基于多维度加权评分的架构选择矩阵。

**评估维度与权重：**

| 维度     | 权重 | 说明                  |
| ------ | -- | ------------------- |
| 任务复杂度  | 5  | 步骤越多、依赖越复杂，得分越高     |
| 实时性要求  | 4  | 响应延迟容忍度越低，得分越高      |
| 多步推理需求 | 4  | 需要前瞻规划和回溯纠错，得分越高    |
| 容错率要求  | 3  | 错误容忍度越低（如金融交易），得分越高 |
| 成本敏感度  | 3  | 预算越紧张，得分越高（低成本倾向）   |

**各架构类型评分（1-10 分）：**

| 维度          | 反射型                  | 规划型                | 协作型                |
| ----------- | -------------------- | ------------------ | ------------------ |
| 任务复杂度（权重5）  | 简单任务：9 / 中等：4 / 复杂：2 | 简单：5 / 中等：8 / 复杂：7 | 简单：3 / 中等：6 / 复杂：9 |
| 实时性要求（权重4）  | 高实时：9 / 中等：6 / 低：3   | 高：4 / 中等：7 / 低：8   | 高：2 / 中等：5 / 低：8   |
| 多步推理需求（权重4） | 无需求：9 / 少量：4 / 大量：1  | 无：6 / 少量：8 / 大量：7  | 无：4 / 少量：6 / 大量：9  |
| 容错率要求（权重3）  | 高容错：8 / 中等：5 / 低容错：2 | 高：6 / 中等：7 / 低：7   | 高：4 / 中等：6 / 低：9   |
| 成本敏感度（权重3）  | 高敏感：9 / 中等：6 / 低：3   | 高：5 / 中等：7 / 低：8   | 高：2 / 中等：5 / 低：8   |

**使用方法：** 根据你的具体场景，在每行选择对应的难度级别，将评分乘以权重后求和，得分最高的架构即为推荐选择。

**计算示例：** 假设场景为"智能客服"——任务复杂度中等、实时性要求高、多步推理需求少量、容错要求中等、成本敏感度高：

```
反射型：4×5 + 9×4 + 4×4 + 5×3 + 9×3 = 20+36+16+15+27 = 114
规划型：8×5 + 4×4 + 8×4 + 7×3 + 5×3 = 40+16+32+21+15 = 124
协作型：6×5 + 2×4 + 6×4 + 6×3 + 2×3 = 30+8+24+18+6  = 86

推荐：规划型（124 分）
```

假设场景为"简单查询路由"——任务简单、高实时、无多步推理、高容错、成本敏感：

```
反射型：9×5 + 9×4 + 9×4 + 8×3 + 9×3 = 45+36+36+24+27 = 168
规划型：5×5 + 4×4 + 6×4 + 6×3 + 5×3 = 25+16+24+18+15 = 98
协作型：3×5 + 2×4 + 4×4 + 4×3 + 2×3 = 15+8+16+12+6  = 57

推荐：反射型（168 分）
```

这个量化矩阵的意义不在于给出"唯一正确答案"，而在于**迫使你系统化地思考每个维度的需求**，避免仅凭直觉选择过于复杂的架构。记住，能用反射型解决的，不要用规划型；能用规划型解决的，不要用协作型。

1. **PDA 循环实现**：选择一种编程语言，实现一个最简的 PDA 循环 Agent。要求：感知阶段接收用户输入，决策阶段调用 LLM 生成下一步行动，行动阶段执行一个预定义的操作（如搜索或计算），循环至少运转3轮。观察并记录：循环的哪一轮产出最有价值的信息？
2. **四组件消融实验**：基于一个现有的 Agent 框架（如 LangChain），设计三组对比实验——(a) 去掉记忆组件，(b) 去掉规划组件，(c) 减少工具数量到1个。用同一个复杂任务（如"调研某技术领域并生成报告"）分别测试，对比三种情况下 Agent 的表现差异。写出你的分析：哪个组件的缺失对结果影响最大？为什么？

## 开放讨论

1. **记忆与遗忘的权衡**：人类的记忆系统有"遗忘机制"——我们会自动淡忘不重要的信息，保留关键记忆。Agent 的记忆系统是否也需要"遗忘"？如果需要，遗忘的标准应该由谁决定（开发者？用户？还是 Agent 自己？）？遗忘可能带来的风险是什么？
2. **单 Agent 的能力极限**：随着 LLM 能力的持续提升（更长的上下文、更强的推理、更丰富的工具），你认为未来单 Agent 是否能承担现在需要多 Agent 才能完成的任务？还是说，多 Agent 架构的某些优势（如角色分工的安全隔离、视角多样性）是单 Agent 无论多强都无法替代的？
3. **协作型架构的"团队病"**：现实中的人类团队经常出现各种协作问题——推诿扯皮、信息孤岛、决策僵局。多 Agent 系统是否也会遇到类似问题？如何设计机制来避免"三个 LLM 开会，讨论了十轮还没结论"的情况？

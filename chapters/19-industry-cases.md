# 第19章 行业案例拆解

> 他山之石，可以攻玉。——《诗经·小雅·鹤鸣》

编程 Agent 是目前 AI Agent 商业化最成熟、最具代表性的领域，客服 Agent 则是企业级应用最广泛的场景，数据分析 Agent 正在改变人与数据的交互方式。拆解这些成功案例的设计思路，不是为了照搬产品形态，而是为了理解"为什么这样设计"——底层能力决定了产品形态，而非反过来。本章将带你理解编程 Agent（Cursor/Claude Code/Devin）的设计思路和核心架构，掌握客服 Agent 从规则引擎到 RAG+工具调用的三代演进，学会数据分析 Agent 的 NL2SQL 和自然语言查询架构，并实战用 LangGraph 复刻简化版编程 Agent。

## 19.1 编程 Agent

2024-2025 年，编程 Agent 从"辅助补全"进化到"自主编码"，深刻改变了开发者的工作方式。拆解这些产品之前需要先明确一点：看到 Cursor 的 Tab 补全或 Devin 的自主编码就想着照搬产品形态，是一个容易踩的坑——这些产品的核心竞争力不在表面功能，而在底层的代码理解能力、上下文管理、安全沙箱等基础设施。如果底层能力没跟上，做出来的东西只会"看起来像但不好用"。更稳妥的做法是先评估自己在代码解析、AST 操作、沙箱执行等基础能力上的成熟度，再决定产品形态的复杂度。

### 19.1.1 Cursor：编辑器原生的智能编程

Cursor 是基于 VS Code 分支改造的 AI 编程编辑器，它的核心设计理念是"AI 能力深度嵌入编辑器，而非外挂"。

**架构分析：**

Cursor Editor Architecture（Cursor 编辑器架构）：

```
+----------------------------------------------+
|                  Cursor Editor                 |
|                                              |
|  +---------+  +---------+  +--------------+ |
|  | Code    |  | File    |  | Terminal/    | |
|  | Editor  |  | Browser |  | Debugger     | |
|  +----┬----+  +----┬----+  +------┬-------+ |
|       |            |               |         |
|  +----v------------v---------------v-------+ |
|  |          Context Engine                   | |
|  |  +----------------------------------+   | |
|  |  | Code Index | Symbol | Deps       |   | |
|  |  +----------------------------------+   | |
|  +----------------┬-----------------------+ |
|                   |                         |
|  +----------------v-----------------------+ |
|  |            AI Core                       | |
|  |  +----------+ +----------+ +--------+ | |
|  |  |Tab Comple.| |Cmd+K Edit| |Chat    | | |
|  |  | (Inline) | |          | | (Ask)  | | |
|  |  +----------+ +----------+ +--------+ | |
|  +----------------┬-----------------------+ |
|                   |                         |
+-------------------┼-------------------------+
                    |
              +-----v------+
              | LLM Service  |
              |(Multi-Model) |
              +--------------+
```

**核心设计模式：**

模式一：上下文优先（Context-First）

Cursor 最突出的设计是上下文引擎。在调用 LLM 之前，Cursor 会自动收集当前文件的上下文：引用的文件、类型定义、相关测试等。用户不需要手动指定上下文，系统自动做"代码 RAG"。

这意味着 Cursor 的 AI 调用不是"裸调用"——每次发给 LLM 的 Prompt 都已经包含了充分的上下文信息。这是它比通用 Chat 更"懂代码"的关键。

模式二：多粒度交互

Cursor 提供了三种不同粒度的 AI 交互：

- Tab 补全：零交互，AI 自动预测你接下来要写的代码，按 Tab 接受。这是最轻量的模式，延迟要求极高（<200ms）。
- Cmd+K 编辑：中等粒度，选中一段代码，用自然语言描述修改意图，AI 在原地生成修改。用户可以接受/拒绝/继续修改。
- Chat 对话：最重量级，自由对话，可以跨文件讨论架构设计，AI 可以读取和修改多个文件。

这三种模式对应了三种不同的 LLM 调用策略：Tab 补全用小模型快速推理，Chat 用大模型深度思考。

模式三：Diff 驱动的代码修改

Cursor 修改代码时，不是直接替换文件内容，而是生成一个 Diff（差异），让用户看到"改了什么"。用户可以逐个接受或拒绝每个修改。这种设计让 AI 的修改是"可审查"的——开发者始终保持对代码的最终控制权。

### 19.1.2 Claude Code：终端里的编程 Agent

Claude Code 是 Anthropic 推出的命令行编程 Agent。和 Cursor 的"编辑器内嵌"路线不同，Claude Code 走的是"终端原生"路线。

**架构分析：**

```
+----------------------------------------------+
|              Claude Code (终端)               |
|                                              |
|  +-----------------------------------------+ |
|  |         Agent Loop (Agent 循环)          | |
|  |                                         | |
|  |  +---------+    +--------------------+  | |
|  |  | 理解任务 |--->| 制定执行计划       |  | |
|  |  +---------+    +--------┬-----------+  | |
|  |                          |              | |
|  |  +-----------------------v-----------+  | |
|  |  |        工具调用循环                |  | |
|  |  |  +------+ +------+ +----------+  |  | |
|  |  |  | 读写  | | 搜索  | | 执行命令 |  |  | |
|  |  |  | 文件  | | 代码  | | (Bash)   |  |  | |
|  |  |  +------+ +------+ +----------+  |  | |
|  |  |                                    |  | |
|  |  |  +------+ +------+ +----------+  |  | |
|  |  |  | Git  | | 网页  | | 创建文件 |  |  | |
|  |  |  | 操作  | | 搜索  | | /目录    |  |  | |
|  |  |  +------+ +------+ +----------+  |  | |
|  |  +-----------------------------------+  | |
|  |                          |              | |
|  |  +-----------------------v-----------+  | |
|  |  |        结果审查与反馈              |  | |
|  |  |  (用户确认 / 自动验证 / 测试运行) |  | |
|  |  +-----------------------------------+  | |
|  +-----------------------------------------+ |
|                                              |
|  +-----------------------------------------+ |
|  |         安全沙箱（权限控制）             | |
|  |  文件读写白名单 / 命令执行确认 / 网络限制| |
|  +-----------------------------------------+ |
+----------------------------------------------+
```

**核心设计模式：**

模式一：Agentic Loop（自主循环）

Claude Code 不是一个简单的"输入-输出"系统，而是一个自主循环的 Agent。它收到任务后，会自己决定下一步做什么：是读文件、搜索代码、执行命令，还是直接写代码。这个循环会一直进行，直到任务完成或需要用户确认。

这种设计的核心优势是：Agent 可以处理需要"多步推理+多工具协作"的复杂任务。比如"给这个项目添加单元测试"，Agent 需要先理解项目结构，再看现有代码，然后决定测试框架，最后逐个编写测试——每一步都依赖上一步的结果。

模式二：工具即是能力边界

Claude Code 的能力完全由其工具集定义。它能读写文件、搜索代码、执行 Bash 命令、操作 Git、搜索网页。每增加一个工具，Agent 的能力就扩展一次。

这给我们一个重要启示：Agent 的能力不取决于 LLM 多聪明，而取决于你给它什么工具。 LLM 是"大脑"，工具是"手脚"，大脑再聪明没有手脚也干不了活。

模式三：人机协作确认

Claude Code 在关键操作前会请求用户确认，特别是：
- 写入文件（特别是覆盖已有文件）
- 执行 Bash 命令
- 删除文件或目录

这种"确认机制"是编程 Agent 的安全底线。AI 可以自主思考，但动手之前要经过人类同意。

### 19.1.3 Devin：自主编程 Agent

Devin 由 Cognition 公司推出，被称为"第一个 AI 软件工程师"。它的设计思路比 Cursor 和 Claude Code 更进一步——追求完全自主。

**架构分析：**

```
+----------------------------------------------------+
|                    Devin 架构                        |
|                                                    |
|  +----------------------------------------------+  |
|  |              规划层 (Planner)                 |  |
|  |  任务分解 → 依赖分析 → 执行计划 → 里程碑定义  |  |
|  +------------------┬---------------------------+  |
|                     |                              |
|  +------------------v---------------------------+  |
|  |            执行层 (Executor)                  |  |
|  |  +--------------------------------------+    |  |
|  |  |         独立沙箱环境                  |    |  |
|  |  |  +--------+ +--------+ +---------+  |    |  |
|  |  |  | 代码编辑| | 浏览器  | |  终端    |  |    |  |
|  |  |  +--------+ +--------+ +---------+  |    |  |
|  |  |  +--------+ +--------+ +---------+  |    |  |
|  |  |  | 文件系统| | Git 仓库| | 运行环境|  |    |  |
|  |  |  +--------+ +--------+ +---------+  |    |  |
|  |  +--------------------------------------+    |  |
|  +------------------┬---------------------------+  |
|                     |                              |
|  +------------------v---------------------------+  |
|  |            验证层 (Verifier)                  |  |
|  |  代码检查 → 测试运行 → 功能验证 → 自我修复   |  |
|  +----------------------------------------------+  |
|                                                    |
|  +----------------------------------------------+  |
|  |            知识层 (Knowledge)                 |  |
|  |  API 文档 | 技术博客 | Stack Overflow | 代码库 |  |
|  +----------------------------------------------+  |
+----------------------------------------------------+
```

**核心设计模式：**

模式一：规划-执行-验证三段式

Devin 将编程过程拆分为三个阶段：规划、执行、验证。这不是简单的"三步走"，而是一个迭代循环——验证不通过时，会回到执行层甚至规划层重新来过。

这种三段式设计是处理复杂任务的关键：没有规划，Agent 会像无头苍蝇一样乱撞；没有验证，Agent 无法知道自己的输出是否正确。

模式二：独立沙箱环境

Devin 运行在一个完全独立的沙箱环境中，拥有自己的文件系统、终端、浏览器和代码编辑器。这意味着 Agent 的操作不会影响用户的本地环境——即使 Agent 写出了破坏性代码，也只是在沙箱里"搞破坏"。

模式三：自主浏览器

Devin 内置了一个自主浏览器，Agent 可以自己浏览网页查找文档、阅读 API 说明、查看 Stack Overflow 上的问答。这是它"自主学习"能力的基础——遇到不懂的问题，它可以自己上网查，而不是只能依赖预置的知识。

### 19.1.4 三款产品的设计启示

对比三款产品，我们可以提炼出编程 Agent 的几个核心设计原则：

| 设计维度 | Cursor | Claude Code | Devin |
|---------|--------|-------------|-------|
| 自主程度 | 低（人主导） | 中（人机协作） | 高（Agent 主导） |
| 上下文获取 | 自动索引 | 按需搜索 | 全环境感知 |
| 安全机制 | Diff 审查 | 确认机制 | 沙箱隔离 |
| 工具范围 | 编辑器内 | 终端+文件+Git | 全开发环境+浏览器 |
| 适用场景 | 日常编码 | 项目开发 | 端到端任务 |

启示一：自主程度和风险成正比

Agent 越自主，效率越高，但风险也越大。Cursor 让人审查每个修改，Devin 让 Agent 自己在沙箱里折腾。选择哪种模式取决于任务的风险容忍度。

启示二：上下文是编程 Agent 的生命线

三款产品都在上下文上投入了大量工程：Cursor 的代码索引、Claude Code 的搜索工具、Devin 的浏览器。给 LLM 足够的上下文，是编程 Agent 成功的前提。

启示三：工具设计决定 Agent 的能力上限

Claude Code 只能做"工具允许的事"。如果工具集里没有"运行测试"，Agent 就永远无法自己验证代码的正确性。设计工具时要思考：Agent 需要什么能力才能完成我的任务？

## 19.2 客服 Agent

客服是 AI Agent 落地最早、最广泛的领域之一。从简单的 FAQ 问答到自主解决复杂问题，客服 Agent 经历了清晰的演进路线。

### 19.2.1 四代客服 Agent 的演进

**第一代：规则引擎（FAQ 匹配）**

```
用户输入 → 关键词匹配 → 返回预置答案
```

这是最原始的"客服 Agent"，本质上就是关键词匹配+预置 FAQ 库。用户说"退货"，系统返回退货政策的文本。优点是简单可控，缺点是完全没有理解能力，换个说法就匹配不上。

**第二代：意图识别 + 槽位填充**

```
用户输入
  |
  v
意图识别（"我想退货" → intent: refund）
  |
  v
槽位填充（订单号？退货原因？）
  |
  v
调用对应业务接口
  |
  v
返回结果
```

第二代引入了 NLU，可以理解用户的意图并提取关键信息（槽位）。这是对话式 AI 的经典架构，被大量智能客服系统采用。

**第三代：RAG 增强 + 工具调用**

```
用户输入
  |
  v
LLM 理解意图
  |
  ├--> RAG 检索知识库 → 补充上下文
  |
  ├--> 工具调用 → 查询订单/物流/库存
  |
  v
LLM 综合生成回答
```

第三代引入了 RAG 和工具调用。Agent 不再只能回答预置的问题，而是可以检索知识库获取最新信息，调用业务系统获取实时数据。

**第四代：自主解决问题**

```
用户输入
  |
  v
Agent 自主规划
  |
  ├--> 分析问题性质
  ├--> 确定解决步骤
  ├--> 调用多个工具（查询 → 判断 → 操作）
  ├--> 主动向用户追问缺失信息
  ├--> 自主决策（如：直接发起退款）
  |
  v
问题解决 + 记录归档
```

第四代 Agent 具备自主规划能力，能处理需要多步操作、跨系统协作的复杂问题。比如用户说"我昨天买的那个东西不要了"，Agent 需要：识别用户身份 → 查找昨天的订单 → 确认商品信息 → 检查退货条件 → 发起退货流程 → 通知用户结果。

### 19.2.3 关键设计决策

决策一：何时转人工

客服 Agent 最关键的设计决策之一是"何时转人工"。转太早，浪费 Agent 的价值；转太晚，用户体验差。常见的策略是：

- 情绪检测：用户情绪激动时自动转人工
- 复杂度评估：Agent 判断自己处理不了时主动转人工
- 金额阈值：涉及金额超过阈值时必须转人工
- 用户请求：用户说"转人工"时立即转

决策二：操作权限分级

客服 Agent 的操作应该分级管控：

| 操作级别 | 示例 | 权限要求 |
|---------|------|---------|
| 只读查询 | 查询订单、物流 | 自动授权 |
| 轻操作 | 修改收货地址、延长收货 | Agent 自主 |
| 重操作 | 发起退款、取消订单 | 用户确认 |
| 敏感操作 | 修改支付方式、大额退款 | 人工审批 |

决策三：知识更新的时效性

客服场景下，知识（退货政策、促销规则等）经常变化。RAG 系统需要保证知识的时效性：

- 热更新：政策变更后立即更新向量库
- 版本管理：保留历史版本，可追溯
- 一致性校验：定期检查知识库和实际政策的一致性

## 19.3 数据分析 Agent

数据分析 Agent 让非技术人员也能用自然语言做数据分析。用户说"上个月各地区的销售额对比"，Agent 自动生成 SQL 查询、执行、画图。

### 19.3.2 核心挑战与解法

挑战一：Schema 理解

数据库的表名和字段名往往是缩写（如 `ord_dtl_amt`），LLM 很难理解。解法是构建一个 Schema 语义层：

```python
SCHEMA_CONTEXT = """
表 orders (订单表):
  - order_id: 订单ID (主键)
  - customer_id: 客户ID (关联 customers 表)
  - region: 地区 (华东/华南/华北/华西/华中)
  - order_date: 下单日期
  - total_amount: 订单总金额 (单位: 元)

表 order_items (订单明细表):
  - item_id: 明细ID (主键)
  - order_id: 订单ID (关联 orders 表)
  - product_name: 商品名称
  - quantity: 购买数量
  - unit_price: 单价 (单位: 元)
"""
```

挑战二：SQL 安全性

用户可能通过自然语言诱导 Agent 生成危险 SQL（如 `DROP TABLE`）。解法是多层防护：

- 白名单机制：只允许 SELECT 查询
- SQL 解析检查：解析生成的 SQL，确认不包含危险操作
- 限制返回行数：默认 LIMIT 1000
- 只读数据库用户：用只读账号执行 SQL

挑战三：结果解读

拿到数据后，Agent 需要自动分析数据并给出洞察，而不仅仅是展示数字。这要求 Agent 具备基本的数据分析能力：计算同比/环比、发现异常值、推荐合适的图表类型。

### 19.3.3 从查询到洞察的完整流程

一个优秀的数据分析 Agent 不是简单的"自然语言转 SQL"工具，而是一个"数据洞察助手"。完整的流程包括：

1. **理解问题**：用户说"上个月销售怎么样"，Agent 需要追问"您关心的是销售额还是订单量？哪个地区？"
2. **生成查询**：根据 Schema 生成正确的 SQL
3. **执行查询**：在数据库上执行，获取结果
4. **数据分析**：计算关键指标、发现趋势、识别异常
5. **可视化**：推荐合适的图表类型，生成可视化
6. **洞察生成**：用自然语言解释数据背后的含义
7. **追问建议**：基于当前分析，建议用户进一步探索的方向

## 19.4 销售 Agent

销售 Agent 是 AI Agent 在商业领域最有价值的应用之一。从线索挖掘到成交跟进，AI 可以在每个环节提升销售效率。

### 19.4.1 销售漏斗与 Agent 的角色

```
+----------------------------------------------------+
|              销售 Agent 全流程架构                    |
|                                                    |
|  +----------+   +----------+   +--------------+   |
|  | 线索挖掘 |-->| 线索评分 |-->|  跟进策略    |   |
|  | Agent    |   | Agent    |   |  规划 Agent  |   |
|  +----------+   +----------+   +------┬-------+   |
|                                       |           |
|  +----------+   +----------+   +-----v--------+  |
|  | 成交分析 |◀--| 谈判辅助 |◀--|  多渠道触达  |  |
|  | Agent    |   | Agent    |   |  Agent       |  |
|  +----------+   +----------+   +--------------+  |
|                                                    |
|  +----------------------------------------------+  |
|  |              数据与知识层                     |  |
|  |  CRM 数据 | 行业报告 | 竞品信息 | 话术库     |  |
|  +----------------------------------------------+  |
+----------------------------------------------------+
```

### 19.4.2 各环节的 Agent 设计

**环节一：线索挖掘 Agent**

线索挖掘 Agent 的核心能力是"从大量信息中找到潜在客户"：

- 监控行业新闻和社交媒体，识别有购买意向的企业
- 分析企业官网和招聘信息，推断其技术栈和需求
- 从公开数据中收集关键联系人信息
- 自动生成线索报告，包含企业画像和推荐触达方式

**环节二：线索评分 Agent**

不是所有线索都值得跟进。线索评分 Agent 根据多个维度给线索打分：

- 企业规模和行业匹配度
- 近期的采购信号（如招聘相关岗位、发布招标信息）
- 与已有客户的相似度
- 预算和决策时间线

**环节三：跟进策略规划 Agent**

根据线索评分和客户画像，规划个性化的跟进策略：

- 选择最佳触达渠道（邮件/电话/微信）
- 生成个性化的开场白和话术
- 规划跟进节奏（何时发第一封邮件、何时打电话、何时拜访）
- 根据客户反馈调整策略

**环节四：多渠道触达 Agent**

执行跟进策略，通过多个渠道触达客户：

- 生成个性化的邮件/消息内容
- 监控客户的打开/回复行为
- 根据客户反馈实时调整沟通策略
- 在合适的时机推动进入下一步

**环节五：谈判辅助 Agent**

在商务谈判阶段提供实时辅助：

- 基于历史成交数据，推荐定价策略
- 分析客户的需求和痛点，建议话术
- 实时生成方案和报价
- 预测成交概率，辅助决策

### 19.4.3 销售 Agent 的关键设计考量

考量一：人机协作模式

销售是高度依赖人际关系的活动，AI 不应替代销售员，而应增强销售员的能力。推荐的设计是"AI 辅助、人类主导"：

- AI 负责信息收集、数据分析、内容生成
- 人类负责关键决策、客户关系维护、谈判临场应变
- AI 提供"第二意见"而非直接决策

考量二：合规与道德

销售 Agent 涉及客户数据，必须考虑合规：

- 数据收集的合法性（GDPR/个保法）
- 自动化触达的合规性（反垃圾邮件法）
- 个性化内容的真实性（不能虚假承诺）
- 客户隐私保护

考量三：效果度量

销售 Agent 的效果应该用业务指标度量：

- 线索到商机的转化率
- 商机到成交的转化率
- 平均成交周期
- 客户满意度
- 销售员的工作效率提升

## 19.5 案例

这一节，我们用 LangGraph 实现一个简化版的编程 Agent，具备代码编写、文件操作和代码执行的能力。

### 19.5.1 设计目标

我们的简化版编程 Agent 应该能够：

1. 理解用户的编程需求
2. 读取和写入文件
3. 执行 Python 代码
4. 查看执行结果并自我修正
5. 完成任务后给出总结

### 19.5.2 状态设计

```python
class AgentState(TypedDict):
    messages: list          # 对话历史
    current_file: str       # 当前操作的文件
    execution_results: list # 执行结果历史
    task_complete: bool     # 任务是否完成
```

### 19.5.3 完整代码

请查看同目录下的 `coding_agent.py` 文件，其中包含完整的编程 Agent 实现。`demo_coding_agent.py` 演示了如何使用这个 Agent。

### 19.5.4 架构解读

```
                    +---------+
                    |  START  |
                    +----┬----+
                         |
                    +----v----+
                    |  Agent  | ◀------------------+
            +-------|  节点   |                    |
            |       +----┬----+                    |
            |            |                         |
            |    +-------v--------+                |
            |    | 路由：选择动作  |                |
            |    +---┬----┬---┬---+                |
            |        |    |   |                    |
            |   +----v+  +v---+ +v-----+          |
            |   |写入 |  |执行| |完成  |          |
            |   |文件 |  |代码| |总结  |---------+
            |   +--┬--+  +-┬--+ +------+
            |      |       |
            |  +---v--+ +--v---+
            |  |读取  | |分析  |
            |  |文件  | |结果  |
            |  +--┬---+ +--┬---+
            |     |        |
            +-----┴--------+
                (回到 Agent 节点)
```

这个架构的核心是 Agent 节点——它负责理解当前状态，决定下一步做什么。路由器根据 Agent 的决策调用不同的工具，工具执行后将结果返回给 Agent，Agent 再决定下一步。

这恰好体现了"他山之石，可以攻玉"——我们从 Cursor、Claude Code 和 Devin 的设计中借鉴了关键思想：Agent 循环、工具集设计、安全沙箱。

## 19.6 进阶拓展

### 19.6.1 多 Agent 协作

本章的案例都是单 Agent 架构。在实际场景中，更常见的模式是多 Agent 协作——每个 Agent 专注于一个领域，通过消息传递协调工作。

例如一个销售系统可以这样设计：

- 线索挖掘 Agent 专注信息收集
- 策略规划 Agent 专注分析和规划
- 执行 Agent 专注触达和沟通
- 监督 Agent 负责质量把控和合规检查

多个 Agent 之间通过共享的状态或消息队列协调，这比一个"万能 Agent"更容易维护和扩展。

### 19.6.2 Agent 的评估框架

如何评估一个 Agent 的好坏？这比评估传统软件困难得多。常用的评估维度包括：

- **任务完成率**：Agent 成功完成任务的比率
- **效率**：完成任务所需的步骤数和时间
- **成本**：完成任务消耗的 Token 和 API 费用
- **鲁棒性**：面对异常输入时的表现
- **用户满意度**：用户对 Agent 表现的主观评价

建议建立一个基准测试集（Benchmark），包含多种典型场景和边界情况，定期运行评估。

### 19.6.3 Agent 的持续改进

Agent 上线后，持续改进的关键是"数据飞轮"：

1. 收集用户反馈（显式评分 + 隐式行为）
2. 分析失败案例（哪些问题 Agent 回答不了）
3. 改进提示词和工具（针对高频失败场景）
4. 扩展知识库（补充缺失的知识）
5. 优化模型选择（根据场景选择最合适的模型）

每一轮改进都会产生新的数据，这些数据又指导下一轮改进——这就是"飞轮效应"。

## 开放讨论

1. **编程 Agent 的边界在哪里**：当前的编程 Agent 已经可以完成大部分编码工作，但在系统架构设计、跨模块协调等高层次任务上仍然力不从心。你认为编程 Agent 的能力天花板在哪里？它会取代程序员，还是会改变程序员的工作方式？

2. **客服 Agent 的信任问题**：用户是否愿意让 Agent 自主执行退款、改单等操作？如果 Agent 犯了错（比如给用户退了不该退的款），责任由谁承担？这些信任和法律问题如何解决？

3. **行业 Agent 的通用性 vs 专用性**：本章讨论的四个行业案例，其 Agent 设计有共同之处（Agent 循环、工具调用、安全机制），但每个行业也有独特的需求。你认为应该构建一个通用 Agent 框架，还是为每个行业定制专用 Agent？
